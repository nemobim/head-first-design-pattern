# 프록시 패턴 (Proxy Pattern)

## 구조

```
Client → Subject(interface) ← Proxy → RealSubject
```

Client는 항상 `Subject` 인터페이스만 바라본다. 실제로 `RealSubject`를 직접 쓰든, `Proxy`를 통해 쓰든 코드가 바뀌지 않는다. `Proxy`는 `RealSubject`와 동일한 인터페이스를 구현하면서, 그 앞에서 접근 제어, 캐싱, 로깅 같은 부가 기능을 수행한다.

- Subject : 공통 인터페이스
- RealSubject : 실제 로직을 처리하는 객체, 어떻게 접근되는지 알 필요없음
- Proxy : RealSubject 앞에서는 대리인

## 프론트엔드에서 프록시가 쓰이는 맥락

### 1. 가상 프록시 (Virtual Proxy) — 지연 로딩

- 무거운 리소스를 실제로 필요할 때까지 로딩을 미루는 패턴. (객체 생성 자체를 미룸)
- 예시) 이미지 레이지 로딩

```tsx
// Subject 인터페이스
interface ImageRenderer {
  render(container: HTMLElement): void;
}

// RealSubject — 실제 이미지 로드 담당
class RealImage implements ImageRenderer {
  private img: HTMLImageElement;

  constructor(private src: string) {
    this.img = new Image();
    this.img.src = src; // 생성 즉시 로드 시작
    console.log(`[RealImage] 로딩 시작: ${src}`);
  }

  render(container: HTMLElement): void {
    container.appendChild(this.img);
  }
}

// Proxy — 뷰포트 진입 전까지 RealImage 생성을 지연
class LazyImageProxy implements ImageRenderer {
  private realImage: RealImage | null = null;
  private placeholder: HTMLDivElement;

  constructor(private src: string) {
    // placeholder만 만들어두고 실제 이미지는 아직 안 만듦
    this.placeholder = document.createElement("div");
    this.placeholder.className = "skeleton";
  }

  render(container: HTMLElement): void {
    container.appendChild(this.placeholder);

    const observer = new IntersectionObserver(([entry]) => {
      if (entry.isIntersecting) {
        // 이 시점에 처음으로 RealImage 생성
        this.realImage = new RealImage(this.src);
        this.realImage.render(container);
        this.placeholder.remove();
        observer.disconnect();
      }
    });

    observer.observe(this.placeholder);
  }
}

// Client 코드 — Proxy인지 Real인지 구분할 필요 없음
function renderGallery(images: string[]) {
  const gallery = document.querySelector(".gallery")!;
  images
    .map((src) => new LazyImageProxy(src))
    .forEach((img) => img.render(gallery));
}
```

> 실무에서는 `<img loading="lazy">`가 브라우저 내장 가상 프록시를 사용하는 것과 같다. 직접 구현해야 할 때는 복잡한 placeholder 전환 로직이 있을 때다.

### 2. 보호 프록시 (Protection Proxy) — 접근 제어

- Java에서는 `InvocationHandler`로 구현, 프론트엔드에서는 권한에 따라 API 호출이나 컴포넌트 접근을 차단하는 식으로 사용
- RealSubject는 누가 접근하는지 알 필요가 없다. 접근 제어는 Proxy 책임
- 예시1) Next.js middleware.ts - 페이지 도달하기전에 실행(인증 토큰 검사)
- 예시2) Route Guard - 라우팅 레벨에서 접근 자체를 차단

```tsx
// Subject — API 서비스 인터페이스
interface UserService {
  getProfile(userId: string): Promise<User>;
  updateProfile(userId: string, data: Partial<User>): Promise<void>;
  deleteAccount(userId: string): Promise<void>;
}

// RealSubject
class ApiUserService implements UserService {
  async getProfile(userId: string) {
    /* API 호출 */
  }
  async updateProfile(userId: string, data: Partial<User>) {
    /* API 호출 */
  }
  async deleteAccount(userId: string) {
    /* API 호출 */
  }
}

// Protection Proxy — 역할(role)에 따라 메서드 접근 차단
class ProtectedUserService implements UserService {
  constructor(
    private service: UserService,
    private currentUser: { role: "admin" | "user"; id: string },
  ) {}

  async getProfile(userId: string) {
    if (this.currentUser.role !== "admin" && this.currentUser.id !== userId) {
      throw new Error("다른 사용자의 프로필을 볼 권한이 없습니다");
    }
    return this.service.getProfile(userId);
  }

  async updateProfile(userId: string, data: Partial<User>) {
    if (this.currentUser.id !== userId) {
      throw new Error("본인 프로필만 수정할 수 있습니다");
    }
    return this.service.updateProfile(userId, data);
  }

  async deleteAccount(userId: string) {
    if (this.currentUser.role !== "admin") {
      throw new Error("계정 삭제는 관리자만 가능합니다");
    }
    return this.service.deleteAccount(userId);
  }
}

// Client는 같은 인터페이스로 호출 — 권한 체크 로직을 몰라도 됨
const service: UserService = new ProtectedUserService(new ApiUserService(), {
  role: "user",
  id: "user-123",
});

await service.deleteAccount("user-456"); // → Error 발생
```

### 3. 캐싱 프록시 (Caching Proxy) — 중복 요청 방지

- 결과 값 재사용. 한 번 수행한 연산의 결과를 저장해두고 같은 요청이 들어오면 RealSubject를 호출하지않고 캐시된 결과 반환
- 실무에서 가장 많이 쓰는 형태
- React Query, SWR 같은 라이브러리가 내부적으로 이 패턴을 구현

```tsx
interface ApiService {
  fetchUser(id: string): Promise<User>;
  fetchPosts(userId: string): Promise<Post[]>;
}

class CachedApiProxy implements ApiService {
  private cache = new Map<string, { data: unknown; expiresAt: number }>();
  private readonly TTL = 5 * 60 * 1000; // 5분

  constructor(private realService: ApiService) {}

  private async withCache<T>(
    key: string,
    fetcher: () => Promise<T>,
  ): Promise<T> {
    const cached = this.cache.get(key);

    if (cached && Date.now() < cached.expiresAt) {
      console.log(`[Cache HIT] ${key}`);
      return cached.data as T;
    }

    console.log(`[Cache MISS] ${key}`);
    const data = await fetcher();
    this.cache.set(key, { data, expiresAt: Date.now() + this.TTL });
    return data;
  }

  fetchUser(id: string) {
    return this.withCache(`user:${id}`, () => this.realService.fetchUser(id));
  }

  fetchPosts(userId: string) {
    return this.withCache(`posts:${userId}`, () =>
      this.realService.fetchPosts(userId),
    );
  }
}
```

## JavaScript `Proxy` 객체 — 언어 내장 프록시

- ES6부터 자바스크립트에 `Proxy` 객체가 내장
  - targe : 원본 객체
  - handler : 동작을 가로채는 객체
  ```tsx
  const proxy = new Proxy(target, handler);
  ```
- 헤드퍼스트의 Java의 `java.lang.reflect.Proxy`, `InvocationHandler`와 거의 동일한 역할을 한다.
  - Java에서는 메서드 호출을 가로채기 위해 동적 프록시를 만들고, 실제 호출 처리를 `InvocationHandler`에게 위임

```tsx
function createValidatedStore<T extends object>(target: T): T {
  return new Proxy(target, {
    set(obj, prop, value) {
      if (prop === "age" && (typeof value !== "number" || value < 0)) {
        throw new RangeError("age는 0 이상의 숫자여야 합니다");
      }
      obj[prop as keyof T] = value;
      console.log(`[Store] ${String(prop)} = ${value}`);
      return true;
    },

    get(obj, prop) {
      console.log(`[Store] ${String(prop)} 접근`);
      return obj[prop as keyof T];
    },
  });
}

const store = createValidatedStore({ name: "지현", age: 28 });
store.age = -1; // → RangeError!
store.name = "이지현"; // → "[Store] name = 이지현"
```

Vue 3의 `reactive()`가 정확하게 동일한 방식으로 동작한다. 프로퍼티 쓰기를 `set` 트랩에서 가로채 컴포넌트 리렌더를 트리거한다.

단, JS `Proxy`와 GoF 프록시 패턴은 완전히 같은 개념이 아니다.

- **GoF Proxy Pattern** — 설계 패턴. 객체 앞에 대리 객체를 두고 접근을 제어하는 구조
- **JavaScript `Proxy`** — 언어 레벨 메타프로그래밍 도구. `get`, `set` 등 기본 동작을 가로챌 수 있음

JS `Proxy`로 GoF 프록시 패턴을 구현할 수 있지만, 둘을 동일한 개념으로 보면 안 된다.

## 헤드퍼스트 개념 → 프론트엔드 대응

| 헤드퍼스트 개념           | 프론트엔드 대응                                         |
| ------------------------- | ------------------------------------------------------- |
| Remote Proxy (RMI)        | `fetch` / `axios` 래퍼 — 네트워크 너머 서버 자원에 접근 |
| Virtual Proxy             | 레이지 로딩 이미지, 코드 스플리팅 (`React.lazy`)        |
| Protection Proxy          | 권한 기반 API 서비스 래퍼, Route Guard, `middleware.ts` |
| Caching Proxy             | React Query, SWR, Apollo Cache                          |
| `java.lang.reflect.Proxy` | JavaScript `Proxy` 객체                                 |
| `InvocationHandler`       | `Proxy`의 `handler` (get / set / apply 트랩)            |

## 핵심 요약

**Decorator vs Proxy**

- Decorator — 기능을 추가한다 ("이걸 덧붙인다")
- Proxy — 접근 자체를 제어한다 ("이 요청을 넘겨도 되는가?")

구조는 비슷하지만 구분 기준은 **의도**다.

> 프록시 패턴의 본질은 "Client는 인터페이스만 알면 되고, 그 뒤에서 무슨 일이 일어나는지 신경 쓸 필요가 없다"는 것이다. 캐싱을 추가하든, 권한 체크를 추가하든, 로딩을 지연하든 — Client 코드는 전혀 바뀌지 않는다.
