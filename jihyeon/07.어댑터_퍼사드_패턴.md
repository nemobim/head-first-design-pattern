# 어댑터 패턴과 퍼사드 패턴

어댑터 패턴(Adapter Pattern)과 퍼사드 패턴(Facade Pattern)은 둘 다 객체나 시스템을 감싸서 사용한다는 점에서 비슷해 보입니다. 하지만 실제로 해결하려는 문제는 다릅니다.

- **어댑터 패턴**은 서로 맞지 않는 인터페이스를 연결하는 데 목적이 있습니다.
- **퍼사드 패턴**은 복잡한 서브시스템을 더 쉽게 사용할 수 있도록 단순한 진입점을 제공하는 데 목적이 있습니다.

즉, 둘 다 래퍼(wrapper)를 사용하지만 초점이 다릅니다.

- 어댑터: **변환**
- 퍼사드: **단순화**

처음 디자인 패턴을 공부할 때 이 둘을 헷갈리기 쉬운데, 아래 질문으로 구분하면 빠릅니다.

- "기존 객체가 있는데 인터페이스가 안 맞는가?" → 어댑터
- "여러 객체를 다뤄야 해서 사용법이 너무 복잡한가?" → 퍼사드

---

## 어댑터 패턴

### 정의

어댑터 패턴은 어떤 클래스의 인터페이스를 클라이언트가 기대하는 다른 인터페이스로 변환하는 패턴입니다.

> 어댑터 패턴은 기존 클래스의 인터페이스를 클라이언트가 원하는 인터페이스로 바꾸어, 서로 호환되지 않는 클래스들이 함께 동작할 수 있게 한다.

### 문제 상황

클라이언트는 `Duck` 타입을 기대하고 있다고 가정해봅시다.

```typescript
interface Duck {
  quack(): void;
  fly(): void;
}
```

그런데 실제로 우리가 가진 객체는 `Turkey`입니다.

```typescript
interface Turkey {
  gobble(): void;
  fly(): void;
}
```

두 인터페이스는 비슷해 보이지만 그대로는 사용할 수 없습니다.

- 오리는 `quack()` 하지만 칠면조는 `gobble()` 합니다.
- 둘 다 `fly()`는 있지만 비행 방식이나 의미가 다를 수 있습니다.
- 클라이언트 코드는 `Duck`만 받도록 설계되어 있습니다.

즉, 기능이 없는 것이 아니라 **호출 방식이 맞지 않는 것**이 문제입니다.

### 해결책

이럴 때 `Turkey`를 `Duck`처럼 보이게 만드는 어댑터를 둡니다.

```typescript
class TurkeyAdapter implements Duck {
  constructor(private turkey: Turkey) {}

  quack(): void {
    this.turkey.gobble();
  }

  fly(): void {
    for (let i = 0; i < 5; i++) {
      this.turkey.fly();
    }
  }
}
```

이제 클라이언트는 `Duck`만 알면 됩니다. 내부에서 실제 작업은 `Turkey`가 수행합니다.

### 구조

어댑터 패턴에는 보통 다음 구성 요소가 있습니다.

- **Client**: 원하는 인터페이스를 사용하는 코드
- **Target**: 클라이언트가 기대하는 인터페이스
- **Adaptee**: 실제로 존재하는 기존 객체
- **Adapter**: `Target` 인터페이스를 구현하면서 내부적으로 `Adaptee`를 호출하는 객체

구조를 간단히 그리면 다음과 같습니다.

```text
Client -> Target
          ^
          |
       Adapter -> Adaptee
```

### 핵심 아이디어

어댑터 패턴의 핵심은 "새 기능을 추가하는 것"이 아닙니다. 이미 존재하는 기능을 **기대하는 형식에 맞게 연결하는 것**입니다.

즉:

- 클라이언트 코드는 유지한다.
- 기존 객체도 유지한다.
- 중간 변환 계층만 추가한다.

이 점 때문에 레거시 코드 통합이나 외부 라이브러리 연동에서 자주 사용됩니다.

### 장점

- 기존 클래스를 수정하지 않고 재사용 가능
- 클라이언트 코드를 변경하지 않고도 호환성 확보 가능
- 레거시 코드와 새 코드를 연결하기 좋음
- 외부 라이브러리 인터페이스를 내부 표준 인터페이스에 맞출 수 있음

### 단점

- 어댑터 클래스가 많아질 수 있음
- 변환 규칙이 복잡하면 코드 추적이 어려워질 수 있음
- 의미 차이가 너무 큰 객체를 억지로 맞추면 설계가 부자연스러워짐

### 언제 사용할까?

- 이미 존재하는 클래스를 다른 인터페이스처럼 사용하고 싶을 때
- 외부 라이브러리를 우리 시스템 인터페이스에 맞추고 싶을 때
- 레거시 코드를 새 구조에 통합해야 할 때
- 클라이언트는 그대로 두고 연결 계층만 바꾸고 싶을 때

## 퍼사드 패턴

### 정의

퍼사드 패턴은 복잡한 서브시스템에 대해 간단한 고수준 인터페이스를 제공하는 패턴입니다.

> 퍼사드 패턴은 복잡한 하위 시스템을 감싸서, 사용하기 쉬운 단일 인터페이스를 제공한다.

### 문제 상황

어떤 기능을 수행하기 위해 여러 객체를 정확한 순서대로 조작해야 하는 경우가 있습니다.

예를 들어 홈시어터로 영화를 본다고 해봅시다. 실제로는 다음과 같은 작업이 필요할 수 있습니다.

- 팝콘 기계 켜기
- 팝콘 튀기기
- 조명 어둡게 하기
- 스크린 내리기
- 프로젝터 켜기
- 앰프 켜기
- 입력 장치 연결하기
- 볼륨 조절하기
- 플레이어 켜기
- 영화 재생하기

이 절차를 클라이언트가 직접 관리하면 문제가 생깁니다.

- 사용법이 복잡함
- 호출 순서를 기억해야 함
- 관련 서브시스템 객체를 모두 알아야 함
- 클라이언트와 하위 시스템 간 결합도가 높아짐
- 실수 가능성이 커짐

### 해결책

이럴 때 퍼사드 객체를 하나 두고, 복잡한 절차를 의미 있는 고수준 메서드로 묶습니다.

```typescript
class HomeTheaterFacade {
  watchMovie(movie: string): void {
    // 내부적으로 여러 장비를 올바른 순서로 동작시킴
  }

  endMovie(): void {
    // 종료 절차를 한 번에 처리
  }
}
```

이제 클라이언트는 세부 절차를 몰라도 됩니다.

```typescript
homeTheater.watchMovie("Raiders of the Lost Ark");
homeTheater.endMovie();
```

### 구조

퍼사드 패턴에는 보통 다음 역할이 있습니다.

- **Client**: 기능을 사용하는 쪽
- **Facade**: 단순한 진입점 제공
- **Subsystem Classes**: 실제 작업을 수행하는 여러 하위 객체들

구조를 간단히 표현하면 다음과 같습니다.

```text
Client -> Facade -> Subsystem A
                 -> Subsystem B
                 -> Subsystem C
```

### 핵심 아이디어

퍼사드 패턴의 핵심은 "숨기는 것" 자체가 아니라, **복잡한 시스템을 더 읽기 쉬운 의도 중심 API로 재구성하는 것**입니다.

예를 들어 클라이언트가 알고 싶은 것은 아래와 같은 수준입니다.

- 영화 보기
- 영화 종료하기
- 라디오 듣기
- 라디오 끄기

반면 실제 시스템은 훨씬 더 저수준의 세부 조작으로 이루어져 있습니다.

퍼사드는 이 간극을 메워줍니다.

### 퍼사드가 모든 것을 감추는 것은 아니다

중요한 점은 퍼사드가 서브시스템 접근을 완전히 차단하는 패턴은 아니라는 것입니다.

- 간단한 사용자는 퍼사드만 써도 된다.
- 고급 사용자는 필요하면 서브시스템을 직접 조작할 수도 있다.

즉, 퍼사드는 **더 쉬운 기본 진입점**을 제공하는 패턴이지, 반드시 모든 저수준 API를 숨기는 패턴은 아닙니다.

### 장점

- 복잡한 시스템을 쉽게 사용할 수 있음
- 클라이언트와 서브시스템 간 결합도를 낮출 수 있음
- 자주 쓰는 절차를 고수준 메서드로 묶을 수 있음
- 사용 흐름을 표준화할 수 있음
- 초보 사용자를 위한 쉬운 API를 제공할 수 있음

### 단점

- 퍼사드에 너무 많은 책임이 몰리면 비대해질 수 있음
- 모든 기능을 퍼사드 하나에 넣으려 하면 오히려 복잡해질 수 있음
- 세부 제어가 필요한 경우 퍼사드만으로는 부족할 수 있음

### 언제 사용할까?

- 여러 객체를 항상 비슷한 순서로 조작해야 할 때
- 라이브러리나 프레임워크의 사용법이 너무 복잡할 때
- 시스템을 쉽게 사용할 수 있는 진입점 API가 필요할 때
- 복잡한 서브시스템 위에 간단한 서비스 계층을 제공하고 싶을 때

## 두 패턴의 차이

처음 배울 때 가장 많이 헷갈리는 부분은 "둘 다 감싸는데 뭐가 다르지?"라는 점입니다. 차이는 감싸는 이유에 있습니다.

| 구분            | 어댑터 패턴                                | 퍼사드 패턴                       |
| --------------- | ------------------------------------------ | --------------------------------- |
| 목적            | 인터페이스 변환                            | 인터페이스 단순화                 |
| 해결 문제       | 호환성 문제                                | 복잡성 문제                       |
| 초점            | 기존 객체를 원하는 형태로 맞춤             | 여러 객체를 쉬운 방식으로 묶음    |
| 대상            | 보통 하나의 객체 또는 인터페이스 차이      | 여러 서브시스템                   |
| 클라이언트 입장 | "이 객체를 내가 원하는 타입처럼 쓰고 싶다" | "이 복잡한 기능을 쉽게 쓰고 싶다" |
| 핵심 동작       | 변환                                       | 조합 및 단순화                    |

- **어댑터는 안 맞는 것을 맞게 만든다.**
- **퍼사드는 복잡한 것을 쉽게 만든다.**

## 언제 무엇을 써야 하나

다음 질문을 기준으로 판단하면 됩니다.

### 1. 기존 객체가 있는데 인터페이스가 맞지 않는가?

이 경우는 어댑터 패턴이 적합합니다.

예:

- 외부 결제 라이브러리의 API를 우리 서비스 인터페이스에 맞추기
- 레거시 로그 시스템을 새로운 로깅 인터페이스에 맞추기
- 서로 다른 포맷의 데이터 제공 객체를 공통 인터페이스로 통일하기

### 2. 여러 객체를 다루는 절차가 너무 복잡한가?

이 경우는 퍼사드 패턴이 적합합니다.

예:

- 여러 인프라 컴포넌트를 한 번에 초기화하는 `ApplicationFacade`
- 복잡한 미디어 장비를 쉽게 다루는 `HomeTheaterFacade`
- 여러 API 호출 절차를 단순하게 제공하는 서비스 계층

### 3. 둘을 함께 쓸 수도 있을까?

그렇습니다. 실제 설계에서는 둘을 함께 쓰는 경우도 많습니다.

예를 들어:

- 시스템 내부에서는 여러 외부 라이브러리를 어댑터로 통일하고
- 그 위에 퍼사드를 올려서 클라이언트가 쉽게 쓰게 만들 수 있습니다.

즉:

- 어댑터는 내부 호환성 문제를 해결하고
- 퍼사드는 외부 사용성 문제를 해결합니다.
