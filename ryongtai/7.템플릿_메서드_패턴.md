# 템플릿 메서드

## 이론 정리

### 1. 템플릿 메서드란?

> “전체 흐름은 내가 정한다. 세부 구현은 네가 채워라.”
> 

템플릿 메서드 패턴의 핵심은,

- 알고리즘의 뼈대를 상위 클래스가 고정하고,
- 각 단계의 구체적인 내용은 하위 클래스가 채우는 것

설계 원칙으로는, 

> 할리우드 원칙(Hollywood Principle)
*”Don’t call us, we’ll call you.”*
> 

하위 클래스는 직접 흐름을 제어하지 않고, 
상위 클래스가 흐름을 쥐고,
필요한 시점에 하위 클래스의 메서드를 호출

즉, 제어권이 위에 있음.

책의 예시에서

```bash
커피 만들기          홍차 만들기
1. 물 끓이기         1. 물 끓이기
2. 커피 우리기       2. 찻잎 우리기
3. 컵에 따르기       3. 컵에 따르기
4. 설탕/크림 추가    4. 레몬 추가
```

1,3번은 같고, 2,4번만 다름

공통의 흐름을 추상 클래스의 `prepareRecipe()` 메서드에 고정하고, 다른 부분만 추상 메서드로 분리

```tsx
abstract class CaffeineBeverage {
  // 템플릿 메서드 — final, 하위 클래스가 바꿀 수 없음
  prepareRecipe() {
    this.boilWater()
    this.brew()           // 추상 메서드
    this.pourInCup()
    this.addCondiments()  // 추상 메서드
  }

  boilWater() { console.log('물 끓이기') }
  pourInCup() { console.log('컵에 따르기') }

  abstract brew(): void
  abstract addCondiments(): void
}

class Coffee extends CaffeineBeverage {
  brew() { console.log('커피 우리기') }
  addCondiments() { console.log('설탕과 크림 추가') }
}
```

여기서 `prepareRecipe()`가 템플릿 메서드.
→ 알고리즘 순서가 여기에 고정

### 2. 훅(Hook) - 선택적 개임

추상 메서드는 반드시 구현해야 하지만, 훅은 선택적

```tsx
abstract class CaffeineBeverage {
  prepareRecipe() {
    this.boilWater()
    this.brew()
    this.pourInCup()
    if (this.customerWantsCondiments()) { // 훅으로 조건 분기
      this.addCondiments()
    }
  }

  // 훅 — 기본 구현이 있고, 필요하면 오버라이드
  customerWantsCondiments(): boolean {
    return true
  }
}

class TeaWithHook extends CaffeineBeverage {
  brew() { console.log('찻잎 우리기') }
  addCondiments() { console.log('레몬 추가') }

  // 이 팀은 컨디먼트 안 원함 — 훅 오버라이드
  customerWantsCondiments() { return false }
}
```

> 훅은 하위 클래스에 “개입할 수 있는 구멍”을 열어줌.
전체 흐름을 바꾸지 않고 특정 시점에만 끼어들 수 있음
>

- “흐름의 고정” 때문
- 상위 클래스가 알고리즘 순서를 `final` (또는 그에 준하는 방식)로 잠가버려야 하위 클래스가 흐름을 바꿀 수 없음.
- 컴포지션으로는 이 “잠금”을 표현하기 어려움

하지만, 상속에는 대가가 있음

```tsx
// 하위 클래스는 상위 클래스의 내부를 알아야 한다
class BadSubclass extends CaffeineBeverage {
  brew() {
    // 상위 클래스의 prepareRecipe가 어떤 순서로 돌아가는지
    // 알고 있어야 여기서 뭘 해야 할지 판단할 수 있다
    // → 강한 결합
  }
}
```

- 하위 클래스가 상위 클래스의 흐름을 “알고”있어야 의미있는 구현이 가능
- 하위 클래스가 상위 클래스에 강하게 묶여있음.
- 상위 클래스 흐름이 바뀌면 하위 클래스 전체가 영향

> 그래서 템플릿 메서드는 클래스 계층이 안정적일 때 써야 함
흐름이 자주 바뀌는 곳에 쓰면 유지보수가 오히려 힘들다.
> 

### 2. 전략 패턴과의 결정적 차이

전략 패턴과 비교하면 템플릿 메서드가 더 명확해짐.
→ 둘 다 “변하는 것을 캡슐화”하는데, 방법이 다름.

- 템플릿 메서드 : 상속으로 해결 (컴파일 타임에 결정)
- 전략 패턴 : 컴포지션으로 해결 (런타임에 교체 가능)

```tsx
// 전략 패턴 — 알고리즘을 객체로 만들어서 주입
class Beverage {
  constructor(private brewStrategy: BrewStrategy) {}
  prepare() {
    this.boilWater()
    this.brewStrategy.brew()  // 런타임에 교체 가능
    this.pourInCup()
  }
}

// 템플릿 메서드 — 상속으로 고정
abstract class Beverage {
  prepare() {           // 흐름 고정
    this.boilWater()
    this.brew()         // 컴파일 타임에 결정됨
    this.pourInCup()
  }
  abstract brew(): void
}
```

실무 판단 기준으로..

- 알고리즘 전체 흐름이 고정이고 일부 단계만 다를 때 → 템플릿 메서드
- 알고리즘 자체를 런타임에 바꿔야 할 때 → 전략 패턴

### 3. 헐리우드 원칙의 더 넓은 의미

책에서 나오는 헐리우드 원칙은, **의존성 역전**과 연결된 개념

- 전통적으로
    - 하위 레벨 코드가 → 상위 레벨 코드를 호출
- 할리우드 원칙은
    - 상위 레벨 코드(추상 클래스)가 → 하위 레벨 코드(구체 클래스)를 호출

> 하위는 상위를 호출하지 않고, 상위가 필요한 시점에 하위 메서드를 부름
→ 제어 흐름 역전
> 

이런 원칙은 템플릿 메서드에서만 나타나는 것은 아님.

```tsx
// React도 헐리우드 원칙으로 작동한다
class MyComponent extends React.Component {
  // 내가 React를 호출하는 게 아니라
  // React가 필요한 시점에 내 메서드를 호출한다
  componentDidMount() { ... }   // React가 호출
  render() { ... }              // React가 호출
}
```

- `render()` 나 `componentDidMount()` 를 내가 직접 부르지 않음.
- React라는 “상위 클래스(프레임워크)”가 흐름을 쥐고 필요한 시점에 호출
- 이것이 템플릿 메서드 패턴이 프레임워크 수준으로 확장된 예시

### 4. 템플릿 메서드의 변형 - 콜백과의 관계

클래스 기반 언어에서는 상속으로 구현하지만,
함수가 일급 객체인 언어(JS/TS)에서는 콜백이나 고차함수로 같은 의도를 표현

```tsx
// 클래스 기반 템플릿 메서드
abstract class DataProcessor {
  process() {
    const raw = this.fetch()
    const parsed = this.parse(raw)
    this.save(parsed)
  }
  abstract fetch(): string
  abstract parse(raw: string): object
  abstract save(data: object): void
}

// 함수형으로 같은 의도 표현
function processData(
  fetch: () => string,
  parse: (raw: string) => object,
  save: (data: object) => void
) {
  const raw = fetch()       // 흐름은 고정
  const parsed = parse(raw)
  save(parsed)
}
```

- 겉 모양은 다르지만 “흐름을 고정하고, 가변 부분만 외부에서 받는다”는 의도는 동일
- 함수형 버전이 더 유연하고(컴파일 타임 결함 없음)
- 클래스 기반 버전이 더 명시적임 (타입, 계약이 클래스에 드러남)

## 활용 예시 - 전통적 프로그래밍 관점

### 1. 프레임워크 생명주기

특히 Java의 `HttpServlet`이 교과서적 예시

```java
// HttpServlet이 템플릿 메서드를 쥐고 있음
public abstract class HttpServlet {
  // 템플릿 메서드 — 흐름 고정
  protected void service(HttpRequest req, HttpResponse res) {
    if (req.getMethod().equals("GET")) {
      doGet(req, res)   // 하위 클래스가 구현
    } else if (req.getMethod().equals("POST")) {
      doPost(req, res)  // 하위 클래스가 구현
    }
    // ... 인증, 로깅, 에러처리 등 공통 흐름
  }

  protected void doGet(HttpRequest req, HttpResponse res) {
    // 기본 구현: 405 Method Not Allowed (훅)
  }
  protected void doPost(HttpRequest req, HttpResponse res) {
    // 기본 구현: 405 Method Not Allowed (훅)
  }
}

// 개발자는 필요한 것만 오버라이드
class UserServlet extends HttpServlet {
  protected void doGet(req, res) { ... }  // GET만 구현
}
```

- `service()` 흐름은 서블릿 컨테이너가 쥐고 있음.
- 개발자는 `doGet`, `doPost`만 채움
- 헐리우드 원칙 그대로

### 2. 정렬 알고리즘

Java의 `Collections.sort()`도 템플릿 메서드 계열
→ 정렬 알고리즘(흐름)은 고정이고,  비교 로직만 외부에서 받음

```java
// 정렬 흐름은 고정, 비교 방법만 주입
Collections.sort(list, (a, b) -> a.getName().compareTo(b.getName()))
```

- 엄밀히는 전략 패턴에 가깝지만, “알고리즘 흐름 고정 + 가변 부분 위임”이라는 템플릿 메서드의 의도를 공유.
- 실무에서는 이 경계가 명확하지 않을 때가 많음

## 활용 예시 - 프론트엔드 생태계

### 1. React 컴포넌트 생명주기 - 패턴의 직접 구현

React 클래스 컴포넌트는 템플릿 메서드 패턴의 프레임워크 수준 구현

```tsx
class React.Component {
  // 템플릿 메서드 — React 내부에 고정된 흐름
  // mount → update → unmount 순서를 React가 관리
  // 개발자는 훅 포인트만 채운다

  componentDidMount() {}    // 훅 — 선택적
  componentDidUpdate() {}   // 훅 — 선택적
  componentWillUnmount() {} // 훅 — 선택적
  render() {}               // 추상 메서드 — 필수
}

class UserProfile extends React.Component {
  componentDidMount() {
    // React가 마운트 완료 후 호출
    fetchUser(this.props.id)
  }
  render() {
    return <div>{this.state.user?.name}</div>
  }
}
```

- `render()`는 추상 메서드(반드시 구현)
- 나머지 생명주기는 훅(선택적 오버라이드)

### 2. 커스텀 훅으로 표현하는 함수형 템플릿 메서드

“데이터 패칭의 흐름”을 고정하고, 가변 부분만 인자로 받는다.

```tsx
// 템플릿 메서드 역할의 커스텀 훅
// 흐름 고정: 로딩 → 패칭 → 성공/에러 처리
function useAsyncData<T>(
  fetcher: () => Promise<T>,          // 추상 메서드에 해당
  onSuccess?: (data: T) => void,      // 훅(선택적)
  onError?: (err: Error) => void      // 훅(선택적)
) {
  const [data, setData] = useState<T | null>(null)
  const [loading, setLoading] = useState(true)
  const [error, setError] = useState<Error | null>(null)

  useEffect(() => {
    setLoading(true)
    fetcher()
      .then(result => {
        setData(result)
        onSuccess?.(result)   // 훅 — 있으면 호출
      })
      .catch(err => {
        setError(err)
        onError?.(err)        // 훅 — 있으면 호출
      })
      .finally(() => setLoading(false))
  }, [])

  return { data, loading, error }
}

// 사용 — fetcher만 바꾸면 됨, 흐름은 고정
const { data: user } = useAsyncData(
  () => apiClient.get('/users/me'),
  (user) => analytics.track('user_loaded', user)  // 훅 활용
)

const { data: orders } = useAsyncData(
  () => apiClient.get('/orders'),
)
```

- `fetcher`는 반드시 줘야 하는 추상 메서드,
- `onSuccess` / `onError`는 선택적 훅
- 패턴 구조가 그대로 함수로 표현

### 3.  폼 처리 흐름 표준화

여러 폼이 있는 서비스에서 “유효성 검사 → 제출 → 피드백” 흐름은 항상 비슷
→ 이 흐름을 고정하고 각 폼의 고유 로직만 주입

```tsx
// 폼 처리 흐름을 고정하는 훅
function useFormTemplate<T>({
  validate,           // 추상 메서드 — 필수
  onSubmit,           // 추상 메서드 — 필수
  onSuccess,          // 훅 — 선택
  onError,            // 훅 — 선택
  transform,          // 훅 — 선택 (제출 전 데이터 변환)
}: FormTemplateOptions<T>) {
  const [isSubmitting, setIsSubmitting] = useState(false)

  const handleSubmit = async (data: T) => {
    // 고정된 흐름
    const errors = validate(data)
    if (errors) return errors

    const payload = transform ? transform(data) : data

    setIsSubmitting(true)
    try {
      await onSubmit(payload)
      onSuccess?.()
    } catch (err) {
      onError?.(err)
    } finally {
      setIsSubmitting(false)
    }
  }

  return { handleSubmit, isSubmitting }
}

// 로그인 폼 — 흐름은 그대로, 고유 로직만 채움
const loginForm = useFormTemplate({
  validate: (data) => loginSchema.safeParse(data),
  onSubmit: (data) => authApi.login(data),
  onSuccess: () => router.push('/dashboard'),
})

// 회원가입 폼 — 같은 흐름, 다른 구현
const signupForm = useFormTemplate({
  validate: (data) => signupSchema.safeParse(data),
  transform: (data) => ({ ...data, createdAt: new Date() }),
  onSubmit: (data) => authApi.signup(data),
  onSuccess: () => toast.success('가입 완료'),
  onError: (err) => toast.error(err.message),
})
```

### 4. Next.js의 페이지 데이터 패칭 패턴

Next.js의 `getServerSideProps` / `generateMetaData` 구조도 템플릿 메서드
→ Next.js가 흐름을 쥐고, 개발자는 지정된 함수만 채움

```tsx
// Next.js가 정한 흐름 (고정):
// 요청 → getServerSideProps 호출 → props 주입 → 렌더링

// 개발자는 이 훅 포인트만 채운다
export async function getServerSideProps(context: GetServerSidePropsContext) {
  const { params, req } = context
  const user = await getUser(req)

  if (!user) {
    return { redirect: { destination: '/login' } }
  }

  return {
    props: { user }
  }
}

// App Router에서도 동일한 의도
export async function generateMetadata({ params }: Props) {
  const product = await getProduct(params.id)
  return {
    title: product.name,
    description: product.description,
  }
}
```

- Next.js라는 프레임워크가 “언제 이 함수를 부를지”를 결정
- 개발자는 “무엇을 반환할지”만 결정

### 마무리 : 프론트엔드에서 이 패턴이 갖는 의미

- 프론트엔드 코드베이스가 커지면 비슷한 흐름이 여러 곳에서 반복됨.
- 데이터 패칭 / 폼 처리 / 인증 체크 / 페이지 초기화
- 이 흐름들이 컴포넌트마다 조금씩 다르게 구현되면 나중에 흐름 하나를 바꿔야 할 때 전체를 뒤져야 함
