# 데코레이터 패턴

## 근본적 문제의식과 핵심 철학

### 근본 문제 : “기능 조합의 폭발”

데코레이터 패턴은 “기능의 동적 조합”이라는 본질적 문제에 대한 구조적 대응
기존 객체의 코드를 수정하지 않으면서도 런타임에 새로운 책임(기능)을 유연하게 덧붙이는 구조를 만드는 것이 데코레이터 패턴 존재의 이유

### 상속으로 기능을 추가한다면?

```typescript
abstract class Beverage {
  abstract getDescription(): string;
  abstract cost(): number;
}

class Espresso extends Beverage {
  getDescription() { return "에스프레소"; }
  cost() { return 3000; }
}

// 에스프레소 + 우유
class EspressoWithMilk extends Beverage {
  getDescription() { return "에스프레소 + 우유"; }
  cost() { return 3500; }
}

// 에스프레소 + 모카
class EspressoWithMocha extends Beverage {
  getDescription() { return "에스프레소 + 모카"; }
  cost() { return 3700; }
}

// 에스프레소 + 우유 + 모카
class EspressoWithMilkAndMocha extends Beverage {
  getDescription() { return "에스프레소 + 우유 + 모카"; }
  cost() { return 4200; }
}

// 에스프레소 + 우유 + 모카 + 휘핑크림
class EspressoWithMilkAndMochaAndWhip extends Beverage {
  // ...이쯤 되면 이미 지옥
}
```
-> 기본 음료가 4종이고, 첨가물이 4종이면 2^4 = 16가지 조합이 가능
-> 거기에 기본 음료 4종을 곱하면…. 64가지 클래스 필요..
-> **클래스 폭발(class explosion)**
-> 그리고 우유 가격이 500원에서 600원으로 바뀌면? 우유가 포함된 모든 클래스를 찾아서 수정해야 함

### 인스턴스 변수로 해결하면?
```typescript
class Beverage {
  description = "";
  milk = false;
  mocha = false;
  whip = false;
  soy = false;

  cost(): number {
    let total = 0;
    if (this.milk) total += 500;
    if (this.mocha) total += 700;
    if (this.whip) total += 400;
    if (this.soy) total += 600;
    return total;
  }
}

class Espresso extends Beverage {
  constructor() {
    super();
    this.description = "에스프레소";
  }

  cost(): number {
    return 3000 + super.cost();
  }
}
```
-> 클래스 폭발은 해결! but..
1. 모든 음료가 첨가물에 대한 지식을 가지고 있어야 함 -> 첨가물이 추가되면,, Berverage Class 수정 -> OCP 위반..
2. 첨가물의 수량은 다룰 수 없음
3. 첨가물이 특정 음료에만 적용 가능한 경우를 처리하기 어려움 (커피류만 샷 추가 등..)
4. 근본적으로.. 이 방식은 **정적**, “기능의 조합”이 객체 내부 상태에 갇혀 있어서 유연성이 떨어짐

### 데코레이터 패턴의 핵심 통찰 : “감싸서 확장하라”

데코레이터 패턴의 철학적 전환점은
> ***”기능을 추가하고 싶어면, 기존 객체를 수정하지 말고 감싸라(wrap)”***

에스프레소에 모카를 추가하고 싶으면,
-> 클래스를 수정하는 것이 아니라 에스프레소 객체를 **~모카라는 껍데기로 감싼다~**
-> 휘핑크림도 추가하고 싶으면 -> **~다시 휘핑크림이라는 껍데기로 감싼다~**

```
   ┌─────────────────────────┐
   │     Whip (400원)         │
   │  ┌───────────────────┐  │
   │  │   Mocha (700원)    │  │
   │  │  ┌─────────────┐  │  │
   │  │  │  Espresso    │  │  │
   │  │  │  (3000원)    │  │  │
   │  │  └─────────────┘  │  │
   │  └───────────────────┘  │
   └─────────────────────────┘
   
   cost() = 400 + 700 + 3000 = 4100원
```

이것을 가능하게 하려면 필요한 핵심 조건이 있음
> ***감싸는 것(데코레이터)과 감싸지는 것(원본)이 같은 타입이어야 함***

### 데코데이터 패턴의 구조적 해결

이를 구현해보면
```typescript
// Component — 원본과 데코레이터의 공통 타입
abstract class Beverage {
  abstract getDescription(): string;
  abstract cost(): number;
}

// ConcreteComponent — 감싸지는 원본 객체
class Espresso extends Beverage {
  getDescription() { return "에스프레소"; }
  cost() { return 3000; }
}

class HouseBlend extends Beverage {
  getDescription() { return "하우스 블렌드"; }
  cost() { return 2500; }
}

// Decorator — 감싸는 껍질의 공통 구조
abstract class CondimentDecorator extends Beverage {
  constructor(protected beverage: Beverage) {
    super();
  }
  // Beverage를 "가지고 있으면서(has-a)" 동시에 Beverage "이다(is-a)"
}

// ConcreteDecorator — 각 첨가물
class Mocha extends CondimentDecorator {
  getDescription() {
    return this.beverage.getDescription() + " + 모카";
  }
  cost() {
    return this.beverage.cost() + 700;
  }
}

class Whip extends CondimentDecorator {
  getDescription() {
    return this.beverage.getDescription() + " + 휘핑크림";
  }
  cost() {
    return this.beverage.cost() + 400;
  }
}

class Milk extends CondimentDecorator {
  getDescription() {
    return this.beverage.getDescription() + " + 우유";
  }
  cost() {
    return this.beverage.cost() + 500;
  }
}
```

사용하는 쪽에서는
```typescript
// 에스프레소 + 모카 2샷 + 휘핑크림
let beverage: Beverage = new Espresso();
beverage = new Mocha(beverage);      // 모카로 감싸기
beverage = new Mocha(beverage);      // 모카 한 번 더 (2샷!)
beverage = new Whip(beverage);       // 휘핑크림으로 감싸기

console.log(beverage.getDescription());  // "에스프레소 + 모카 + 모카 + 휘핑크림"
console.log(beverage.cost());            // 3000 + 700 + 700 + 400 = 4800
```

새로운 첨가물을 추가하려면? `CondimentDecorator`를 상속한 클래스를 하나 만들면 됨.
기존의 어느 클래스도 수정하지 않음

-> 감싸는 순서와 조합을 코드로 동적으로 작성해 런타임에 조합을 바꿀 수 있음.

## OCP - 패턴의 핵심
> 개방-폐쇄 원칙(Open-Closed Principle, OCP)
> -> *“클래스는 확장에는 열려 있어야 하지만, 변경에는 닫혀 있어야 한다.”*
(데코레이터 패턴에서 가장 극적으로 드러남)

- 확장에 열려있다 -> 데코레이터를 만들어 기능을 추가할 수 있음
- 변경에 닫혀있다 -> 기존 클래스를 수정할 필요가 없다.

즉,
- 상속을 통한 확장은 -> 코드를 물려받아서 수정하는 것
- 데코레이터를 통한 확장 -> 기존 코드를 건드리지 않고 바깥에서 감싸는 것
-> 이 차이가 OCP를 실현하는가 위반하는가를 가름.


### 데코레이터 패턴의 독특한 구조적 특성 : *is-a*와 *has-a*의 공존

전략 패턴은 “상속보다 구성을 활용하라” -> ***~is-a 대신 has-a를 쓰라는 것~***
하지만 데코레이터 패턴은 ~***is-a와 has-a를 동시에 사용***~

```typescript
abstract class CondimentDecorator extends Beverage {   // is-a: Beverage 타입
  constructor(protected beverage: Beverage) {           // has-a: Beverage를 감싸고 있음
    super();
  }
}
```
-> 여기서의 상속(is-a)은 “행동을 물려받자”가 아니라, 원본과 타입을 갖게 만들어 중첩이 가능하게 하기 위함
-> 실제 행동은 구성(has-a)을 통해 내부에 감싸고 있는 객체에 위임

즉, 
- 데코레이터에서의 상속은 “행동의 재사용”이 아니라,
- ”타입의 호환”을 위한 것
- 실제 행동은 여전히 구성(has-a)를 통해 이뤄짐

### 앞선 패턴들과의 관계 : 같은 원칙, 다른 적용

- 전략 패턴 -> 
  - “행동을 교체”, 
  - 하나의 행동을 다른 행동으로 바꿈 (A -> B), 
  - 수평적 교체
- 옵저버 패턴 -> 
  - “변경을 전파”, 
  - 하나의 변경이 여러곳에 영향을 미칠 때 그 전파를 구조화
- 데코레이터 패턴 -> 
  - “기능을 누적”, 
  - 기존 기능 위에 새로운 기능을 쌓음 (A+B)
  - 수직적 누적

### 더 넓은 맥락에서의 데코레이터 패턴
데코레티어 패턴도
> 기존 코드를 건드리지 않고 기능을 확장하는 방법

소프트웨어 설계 전반에서 
- 미들웨어 체인
- 고차 함수
- Higher-Order Component
- 파이프라인 패턴
등의 형태로 나타남 -> “기존의 것을 감싸서 기능을 덧붙인다.”는 동일한 원리

프론트엔드에서는
- 함수형 프로그래밍의 “함수 합성”이라는 개념으로 이어짐
- ’요청 -> 가공 -> 응답’ 이라는 파이프라인 구조가 많아 패턴 활용이 많음.
  - HTTP 요청에 인증 헤더 붙이기 / 로깅 추가하기 / 에러 핸들링 감사기
  - 컴포넌트에 로딩 상태 추가 / 에러 바운더리 / 권한 체크

## 실전 - 전통적인 프로그래밍 관점에서

### 데코레이터 패턴이 없다면?
> 하나의 핵심 동작에 부가 기능을 선택적으로 덧붙여야 하는 상황

로깅 시스템
```java
class Logger {
    public void log(String message) {
        System.out.println(message);
    }
}

// 타임스탬프가 필요한 곳
class TimestampLogger extends Logger {
    public void log(String message) {
        System.out.println("[" + Instant.now() + "] " + message);
    }
}

// JSON 형식이 필요한 곳
class JsonLogger extends Logger {
    public void log(String message) {
        System.out.println("{\"message\": \"" + message + "\"}");
    }
}

// 타임스탬프 + JSON이 필요한 곳
class TimestampJsonLogger extends Logger { ... }

// 타임스탬프 + JSON + 파일 출력이 필요한 곳
class TimestampJsonFileLogger extends Logger { ... }

// 타임스탬프 + 암호화가 필요한 곳
class TimestampEncryptedLogger extends Logger { ... }

// 😱 기능이 5개면 2⁵ = 32개 클래스...
```

### 적용한다면? : 기능을 겹겹이 감싸기
```java
// Component 인터페이스
interface Logger {
    void log(String message);
}

// ConcreteComponent — 핵심 동작
class ConsoleLogger implements Logger {
    public void log(String message) {
        System.out.println(message);
    }
}

class FileLogger implements Logger {
    public void log(String message) {
        // 파일에 기록
        Files.writeString(logFile, message + "\n", APPEND);
    }
}

// Decorator 기반 클래스
abstract class LoggerDecorator implements Logger {
    protected final Logger wrapped;
    
    LoggerDecorator(Logger wrapped) {
        this.wrapped = wrapped;
    }
}

// ConcreteDecorators — 각 부가 기능
class TimestampDecorator extends LoggerDecorator {
    TimestampDecorator(Logger wrapped) { super(wrapped); }
    
    public void log(String message) {
        String timestamped = "[" + Instant.now() + "] " + message;
        wrapped.log(timestamped);  // 변환된 메시지를 안쪽으로 위임
    }
}

class JsonDecorator extends LoggerDecorator {
    JsonDecorator(Logger wrapped) { super(wrapped); }
    
    public void log(String message) {
        String json = "{\"message\": \"" + message + "\", \"timestamp\": \"" + Instant.now() + "\"}";
        wrapped.log(json);
    }
}

class EncryptionDecorator extends LoggerDecorator {
    EncryptionDecorator(Logger wrapped) { super(wrapped); }
    
    public void log(String message) {
        String encrypted = encrypt(message);
        wrapped.log(encrypted);
    }
}

class UpperCaseDecorator extends LoggerDecorator {
    UpperCaseDecorator(Logger wrapped) { super(wrapped); }
    
    public void log(String message) {
        wrapped.log(message.toUpperCase());
    }
}
```

사용하는 쪽에서 자유롭게 조합
```java
// 콘솔 + 타임스탬프
Logger logger1 = new TimestampDecorator(new ConsoleLogger());

// 파일 + 타임스탬프 + 암호화
Logger logger2 = new EncryptionDecorator(new TimestampDecorator(new FileLogger()));

// 콘솔 + JSON + 대문자
Logger logger3 = new UpperCaseDecorator(new JsonDecorator(new ConsoleLogger()));

// 모든 조합이 가능하되, 클래스는 기능 개수만큼만 존재
// 5개 기능 → 5개 데코레이터 클래스 (32개가 아니라!)
```

## 실전 - 프론트엔드 생태계 관점에서

### 프론트엔드에서 데코레이터 패턴이 특별한 이유
- 전통적인 클래스 래핑 형태로 잘 나타나지 않음
- 주로, 함수 래핑 / 미들웨어 / HOC / 함수 합성
  - JS의 일급 함수 특성 때문

### 1. 함수 데코레이터
```typescript
// 원본 함수를 감싸서 기능을 추가하는 데코레이터들
type AnyFn = (...args: any[]) => any;

// 실행 시간 측정 데코레이터
function withTiming<T extends AnyFn>(fn: T, label?: string): T {
  return function (this: any, ...args: any[]) {
    const start = performance.now();
    const result = fn.apply(this, args);
    const duration = performance.now() - start;
    console.log(`[${label ?? fn.name}] ${duration.toFixed(2)}ms`);
    return result;
  } as T;
}

// 로깅 데코레이터
function withLogging<T extends AnyFn>(fn: T, label?: string): T {
  return function (this: any, ...args: any[]) {
    console.log(`[CALL] ${label ?? fn.name}`, args);
    const result = fn.apply(this, args);
    console.log(`[RETURN] ${label ?? fn.name}`, result);
    return result;
  } as T;
}

// 에러 핸들링 데코레이터
function withErrorHandling<T extends AnyFn>(fn: T, fallback?: any): T {
  return function (this: any, ...args: any[]) {
    try {
      return fn.apply(this, args);
    } catch (error) {
      console.error(`Error in ${fn.name}:`, error);
      return fallback;
    }
  } as T;
}

// 원본 함수
function calculateTotal(items: { price: number }[]): number {
  return items.reduce((sum, item) => sum + item.price, 0);
}

// 데코레이터로 감싸기 — 커피숍의 마트료시카와 동일한 구조
const enhancedCalculate = withTiming(
  withLogging(
    withErrorHandling(calculateTotal, 0)
  )
);

// 원본 함수는 전혀 수정되지 않았다
// 실행하면: 로깅 → 에러 핸들링 → 원본 실행 → 타이밍 측정
enhancedCalculate([{ price: 1000 }, { price: 2000 }]);
```
-> 원본 함수 수정이 없고, 어떤 데코레이터로 감싸든 원본의 동작은 보존

### 2. 미들웨어 패턴 - 데코레이터 체인
Express의 미들웨어는 데코레이터 패턴이 체인으로 확장된 형태
```typescript
// Express 미들웨어 = 요청/응답 처리를 감싸는 데코레이터
// 각 미들웨어는 독립적이고, 순서대로 중첩되어 실행됨

app.use(cors());              // CORS 헤더 추가 — 데코레이터 1
app.use(helmet());            // 보안 헤더 추가 — 데코레이터 2
app.use(morgan('combined'));  // 요청 로깅    — 데코레이터 3
app.use(compression());      // 응답 압축     — 데코레이터 4
app.use(authMiddleware);      // 인증 검증     — 데코레이터 5

// 실제 핸들러 = ConcreteComponent (원본)
app.get('/api/data', (req, res) => {
  res.json({ data: 'hello' });
});

// 요청 흐름:
// cors → helmet → morgan → compression → auth → 핸들러 → (역순으로 응답)
```

```typescript
type Middleware<T> = (context: T, next: () => void) => void;

function compose<T>(middlewares: Middleware<T>[]) {
  return (context: T) => {
    let index = 0;

    function next() {
      if (index < middlewares.length) {
        const middleware = middlewares[index++];
        middleware(context, next);
      }
    }

    next();
  };
}

// 각 미들웨어는 독립적인 데코레이터
const timing: Middleware<Request> = (ctx, next) => {
  const start = Date.now();
  next();  // 안쪽(다음 미들웨어 또는 원본)으로 위임
  console.log(`${Date.now() - start}ms`);
};

const auth: Middleware<Request> = (ctx, next) => {
  if (!ctx.headers.authorization) throw new Error('Unauthorized');
  next();  // 인증 통과 시 안쪽으로 위임
};

const handler = compose([timing, auth, /* ...더 많은 미들웨어 */]);
```

### 4. Higher-Order Component (HOC) - React!
컴포넌트를 받아서 기능이 추가된 새 컴포넌트를 반환
```tsx
// 데코레이터 1: 로딩 상태 추가
function withLoading<P extends object>(
  WrappedComponent: React.ComponentType<P>,
) {
  return function WithLoadingComponent(props: P & { isLoading: boolean }) {
    const { isLoading, ...rest } = props;
    if (isLoading) return <Spinner />;
    return <WrappedComponent {...(rest as P)} />;
  };
}

// 데코레이터 2: 에러 바운더리 추가
function withErrorBoundary<P extends object>(
  WrappedComponent: React.ComponentType<P>,
) {
  return class WithErrorBoundary extends React.Component<P, { hasError: boolean }> {
    state = { hasError: false };
    static getDerivedStateFromError() { return { hasError: true }; }
    render() {
      if (this.state.hasError) return <ErrorFallback />;
      return <WrappedComponent {...this.props} />;
    }
  };
}

// 데코레이터 3: 권한 체크 추가
function withAuth<P extends object>(
  WrappedComponent: React.ComponentType<P>,
  requiredRole: string,
) {
  return function WithAuthComponent(props: P) {
    const { user } = useAuth();
    if (!user) return <Navigate to="/login" />;
    if (!user.roles.includes(requiredRole)) return <Forbidden />;
    return <WrappedComponent {...props} />;
  };
}

// 원본 컴포넌트 — 순수하게 UI만 담당
function UserDashboard({ users }: { users: User[] }) {
  return (
    <ul>
      {users.map(user => <li key={user.id}>{user.name}</li>)}
    </ul>
  );
}

// 데코레이터 중첩 — 커피숍의 마트료시카와 동일
const EnhancedDashboard = withErrorBoundary(
  withAuth(
    withLoading(UserDashboard),
    'admin',
  )
);

// 사용하는 쪽은 EnhancedDashboard가 내부적으로
// 에러 처리 + 권한 체크 + 로딩 처리를 하는지 모른다
<EnhancedDashboard users={users} isLoading={loading} />
```

- `UserDashboard` : ConcreteComponent
- `WithLoading` / `withAuth` / `withErrorBoundary` : Decorator
- 모두 `React.ComponentType` 이라는 같은 타입을 공유!
- 요즘은 커스텀 훅으로 대체되는 추세 (prop 충돌, wrapper hell)

### 5. 커스텀 훅
HOC가 컴포넌트를 감싸는 데코레이터라면, 커스텀 훅은 로직을 감싸는 데코레이터
```tsx

function useUsers() {
  return useQuery({
    queryKey: ['users'],
    queryFn: () => api.get('/users'),
  });
}

function useAuthGuard(requiredRole: string) {
  const { user } = useAuth();
  const navigate = useNavigate();

  useEffect(() => {
    if (!user) navigate('/login');
  }, [user, navigate]);

  return { isAuthorized: user?.roles.includes(requiredRole) ?? false };
}

  useEffect(() => {
    document.title = title;
  }, [title]);
}

// 컴포넌트에서 훅을 조합 — HOC의 감싸기 대신 훅의 나열
function UserDashboard() {
  const { isAuthorized } = useAuthGuard('admin');   // 기능 1: 권한 체크
  const { data, isLoading, error } = useUsers();    // 기능 2: 데이터 로딩
  useDocumentTitle('사용자 관리');                     // 기능 3: 문서 타이틀

  if (!isAuthorized) return <Forbidden />;
  if (isLoading) return <Spinner />;
  if (error) return <ErrorMessage error={error} />;

  return <UserList users={data} />;
}
```

### 6. fetch/API 래퍼
```typescript
type FetchFn = typeof fetch;

// 데코레이터 1: 기본 설정 추가
function withBaseURL(fetchFn: FetchFn, baseURL: string): FetchFn {
  return (input, init) => {
    const url = typeof input === 'string' ? `${baseURL}${input}` : input;
    return fetchFn(url, init);
  };
}

// 데코레이터 2: 인증 헤더 주입
function withAuth(fetchFn: FetchFn, getToken: () => string): FetchFn {
  return (input, init) => {
    const headers = new Headers(init?.headers);
    headers.set('Authorization', `Bearer ${getToken()}`);
    return fetchFn(input, { ...init, headers });
  };
}

// 데코레이터 3: 자동 JSON 파싱
function withJsonParsing(fetchFn: FetchFn): FetchFn {
  return async (input, init) => {
    const response = await fetchFn(input, {
      ...init,
      headers: { 'Content-Type': 'application/json', ...init?.headers as Record<string, string> },
    });
    return response;
  };
}

// 데코레이터 4: 재시도
function withRetry(fetchFn: FetchFn, maxRetries = 3): FetchFn {
  return async (input, init) => {
    let lastError: Error | null = null;
    for (let i = 0; i <= maxRetries; i++) {
      try {
        return await fetchFn(input, init);
      } catch (error) {
        lastError = error as Error;
        if (i < maxRetries) await new Promise(r => setTimeout(r, 2 ** i * 1000));
      }
    }
    throw lastError;
  };
}

// 데코레이터 5: 요청/응답 로깅
function withLogging(fetchFn: FetchFn): FetchFn {
  return async (input, init) => {
    const url = typeof input === 'string' ? input : input.toString();
    console.log(`[→] ${init?.method ?? 'GET'} ${url}`);
    const start = Date.now();
    const response = await fetchFn(input, init);
    console.log(`[←] ${response.status} ${url} (${Date.now() - start}ms)`);
    return response;
  };
}

// 조합: 마트료시카처럼 감싸기
const apiFetch = withLogging(
  withRetry(
    withAuth(
      withJsonParsing(
        withBaseURL(fetch, 'https://api.example.com')
      ),
      () => localStorage.getItem('token') ?? '',
    ),
  )
);

// 사용하는 쪽은 모든 부가 기능을 의식하지 않음
const response = await apiFetch('/users');
```

### 7. pipe/compose - FP
```typescript
// compose: 오른쪽에서 왼쪽으로 합성 (데코레이터 중첩 순서와 동일)
function compose<T>(...fns: ((arg: T) => T)[]): (arg: T) => T {
  return (arg: T) => fns.reduceRight((acc, fn) => fn(acc), arg);
}

// pipe: 왼쪽에서 오른쪽으로 합성 (읽기 순서와 동일)
function pipe<T>(...fns: ((arg: T) => T)[]): (arg: T) => T {
  return (arg: T) => fns.reduce((acc, fn) => fn(acc), arg);
}

// 문자열 가공 데코레이터들
const trim = (s: string) => s.trim();
const lowercase = (s: string) => s.toLowerCase();
const removeSpecialChars = (s: string) => s.replace(/[^a-z0-9\s가-힣]/g, '');
const replaceSpaces = (s: string) => s.replace(/\s+/g, '-');

// pipe로 데코레이터 체인 구성 — 가독성이 훨씬 좋다
const slugify = pipe(
  trim,
  lowercase,
  removeSpecialChars,
  replaceSpaces,
);

slugify("  Hello World! 안녕  ");  // "hello-world-안녕"

// 새 단계 추가? pipe에 함수 하나만 추가하면 끝
const slugifyWithMaxLength = pipe(
  trim,
  lowercase,
  removeSpecialChars,
  replaceSpaces,
  (s: string) => s.slice(0, 50),  // 50자 제한 추가
);
```

- pipe는 데코레이터 패턴의 중첩 구조를 평면적인 리스트로 바꿔줌

```typescript
function withStaleTime<T extends object>(options: T, time: number) {
  return { ...options, staleTime: time };
}

function withRefetchInterval<T extends object>(options: T, interval: number) {
  return { ...options, refetchInterval: interval };
}

function withRetry<T extends object>(options: T, count: number) {
  return { ...options, retry: count, retryDelay: (attempt: number) => 2 ** attempt * 1000 };
}

function withPlaceholder<T extends object>(options: T, data: unknown) {
  return { ...options, placeholderData: data };
}

// 조합: 필요한 기능만 선택적으로 덧붙임
const enhancedQuery = withRetry(
  withStaleTime(
    withRefetchInterval(baseQuery, 30000),
    5 * 60 * 1000,
  ),
  3,
);

// pipe로 더 깔끔하게
const enhancedQuery2 = pipe(
  (q) => withStaleTime(q, 5 * 60 * 1000),
  (q) => withRefetchInterval(q, 30000),
  (q) => withRetry(q, 3),
  (q) => withPlaceholder(q, []),
)(baseQuery);

const { data } = useQuery(enhancedQuery2);
```
