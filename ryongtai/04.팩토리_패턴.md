# 팩토리 패턴

## 1. 근본적 문제의식과 핵심 철학

### 근본 문제 : ‘객체 생성의 결합’

소프트웨어에서 객체를 사용하려면 먼저 생성해야 함

→ 생성(`new` )는 본질적으로 구체 클래스에 대한 의존을 만듬

> “객체를 생성하는 행위가 만들어내는 결합” 이라는 구조적 문제 발생
> 

앞의 패턴들을 되짚어보면,

> “구현이 아닌 인터페이스에 맞춰 프로그래밍하라”
> 

였지만, `new` 를 쓰는 순간 인터페이스가 아니라 구현 클래스를 직접 지정하고 있기 때문에 이 원칙이 위반.

> 객체를 “사용하는” 코드에서 객체를 “생성하는” 코드를 분리하는 것 → 팩터리 패턴
> 

### new의 문제 : 구상 클래스에 대한 의존

피자가게 예시

```typescript
class PizzaStore {
  orderPizza(type: string): Pizza {
    let pizza: Pizza;

    // 💣 이 부분이 문제다
    if (type === "cheese") {
      pizza = new CheesePizza();
    } else if (type === "pepperoni") {
      pizza = new PepperoniPizza();
    } else if (type === "clam") {
      pizza = new ClamPizza();
    } else if (type === "veggie") {
      pizza = new VeggiePizza();
    } else {
      throw new Error(`Unknown pizza type: ${type}`);
    }

    // 이 부분은 변하지 않는다
    pizza.prepare();
    pizza.bake();
    pizza.cut();
    pizza.box();

    return pizza;
  }
}
```

→ 변하는 부분과 변하지 않는 부분이 혼재

> 앞선 원칙 “변하는 부분을 찾아서 캠슐화하라”
> 
> 
> → 이 코드에서 변하는 부분은? 객체를 생성하는 코드!
> 

하지만, 단순히 `orderPizza()` 에서만 발생하는 문제는 아닐 수 있음.

예를 들어, 메뉴 표시 / 가격 계산 / 재고 관리 등에서도 동일한 `if-else` 생성 코드가 중복될 수 있음.

→ 한 곳에서 피자 종류를 수정하면 모든 곳에서 수정해야 하고, 빼먹으면 문제 발생!

### 해결 방법은? - 심플 팩토리 (Simple Factory)

가장 직관적인 해결은 객체 생성 코드를 별도의 클래스로 추출하는 것

엄밀한 의미에서 디자인 패턴이라기 보단,

→ 프로그래밍 관용구 (idiom)

```typescript
// 생성 로직을 한 곳으로 모음
class SimplePizzaFactory {
  createPizza(type: string): Pizza {
    if (type === "cheese") return new CheesePizza();
    if (type === "pepperoni") return new PepperoniPizza();
    if (type === "clam") return new ClamPizza();
    if (type === "veggie") return new VeggiePizza();
    throw new Error(`Unknown pizza type: ${type}`);
  }
}

class PizzaStore {
  constructor(private factory: SimplePizzaFactory) {}

  orderPizza(type: string): Pizza {
    // 생성은 팩토리에 위임
    const pizza = this.factory.createPizza(type);

    // 변하지 않는 흐름만 남음
    pizza.prepare();
    pizza.bake();
    pizza.cut();
    pizza.box();

    return pizza;
  }
}
```

→ 더 이상 `PizzaStore`는 `CheesePizza` 라는 이름을 모른다!

→ 피자 종류에 변화가 생기면 `SimplePizzaFactory`만 수정! → 다른 코드는 영향x

하지만, 이 심플 팩토리는 한계가 있음.

피자 종류가 다양/복잡해지면 이 심플 팩토리 안의 `if-else` 가 다시 폭발

### 다른 해결은? - 팩토리 메서드 패턴 (Factory Method Pattern)

> ***“객체 생성을 서브클래스에게 맡겨라”***
> 

**심플 팩토리**에서는 별도의 팩토리 객체가 생성을 담당

**팩토리 메서드 패턴**에서는 생성 메서드를 추상 메서드로 선언하고, 서브클래스가 구현을 담당!

→ ***“이 메서드를 오버라이드해서 네가 원하는 객체를 만들어라”***

```typescript
// Creator — 추상 클래스
abstract class PizzaStore {
  // 팩토리 메서드: 서브클래스가 구현
  abstract createPizza(type: string): Pizza;

  // 변하지 않는 흐름 (템플릿)
  orderPizza(type: string): Pizza {
    const pizza = this.createPizza(type);  // 서브클래스에게 위임

    pizza.prepare();
    pizza.bake();
    pizza.cut();
    pizza.box();

    return pizza;
  }
}

// ConcreteCreator — 뉴욕 스타일
class NYPizzaStore extends PizzaStore {
  createPizza(type: string): Pizza {
    if (type === "cheese") return new NYStyleCheesePizza();
    if (type === "pepperoni") return new NYStylePepperoniPizza();
    if (type === "clam") return new NYStyleClamPizza();
    throw new Error(`Unknown type: ${type}`);
  }
}

// ConcreteCreator — 시카고 스타일
class ChicagoPizzaStore extends PizzaStore {
  createPizza(type: string): Pizza {
    if (type === "cheese") return new ChicagoStyleCheesePizza();
    if (type === "pepperoni") return new ChicagoStylePepperoniPizza();
    if (type === "clam") return new ChicagoStyleClamPizza();
    throw new Error(`Unknown type: ${type}`);
  }
}

// 사용
const nyStore = new NYPizzaStore();
const pizza = nyStore.orderPizza("cheese");  // NYStyleCheesePizza가 만들어짐

const chicagoStore = new ChicagoPizzaStore();
const pizza2 = chicagoStore.orderPizza("cheese");  // ChicagoStyleCheesePizza가 만들어짐
```

- `orderPizza()`의 흐름이 부모 클래스에서 고정 (prepare → bake → cut → box)
    - 모든 피자 가게가 동일한 프로세스를 따름
    - 하지만, “어떤 피자를 만들 것인가”는 각 서브클래스가 결정

즉, **프로세스는 상위에서 정의**하고, **생성의 결정은 하위에 위임**

→ *“구현이 아닌 인터페이스에 맞춰 프로그래밍하라”*는 원칙이 객체 생성 단계에까지 확장된 

### 또 다른 해결은? - 추상 팩토리 패턴 (Abstract Factory Pattern)

**팩토리 메서드 패턴**이 *“어떤 제품을 만들 것인가”*의 결정을 위임한다면,

**추상 팩토 패턴**은 *“관련 있는 제품군 전체를 어떻게 만들 것인가”*를 위임

책의 예시에서

- 피자 하나를 만들려면 도우/치즈/소스/토핑이라는 여러 재료가 필요
- 뉴욕 피자와 시카고 피자는 재료가 전부 다름
    - 뉴욕 → 얇은 도우 + 마리나라 소스 + 레지아노 치즈
    - 시카고 → 두꺼운 도우 + 토마토 소스 + 모짜렐라 치즈
- **이런 재료군 전체를 일관되게 생성하는 것이 추상 팩토리의 역할**

```typescript
// Abstract Factory — 재료군 생성의 인터페이스
interface PizzaIngredientFactory {
  createDough(): Dough;
  createSauce(): Sauce;
  createCheese(): Cheese;
  createToppings(): Topping[];
}

// Concrete Factory — 뉴욕 재료군
class NYPizzaIngredientFactory implements PizzaIngredientFactory {
  createDough() { return new ThinCrustDough(); }
  createSauce() { return new MarinaraSauce(); }
  createCheese() { return new ReggianoCheese(); }
  createToppings() { return [new Garlic(), new Onion(), new Mushroom()]; }
}

// Concrete Factory — 시카고 재료군
class ChicagoPizzaIngredientFactory implements PizzaIngredientFactory {
  createDough() { return new ThickCrustDough(); }
  createSauce() { return new PlumTomatoSauce(); }
  createCheese() { return new MozzarellaCheese(); }
  createToppings() { return [new BlackOlives(), new Spinach(), new Eggplant()]; }
}

// Pizza는 어떤 재료 팩토리가 주입되느냐에 따라 달라짐
class CheesePizza extends Pizza {
  constructor(private ingredientFactory: PizzaIngredientFactory) {
    super();
  }

  prepare() {
    this.dough = this.ingredientFactory.createDough();
    this.sauce = this.ingredientFactory.createSauce();
    this.cheese = this.ingredientFactory.createCheese();
    // CheesePizza는 재료의 구체적 종류를 모른다
    // 팩토리가 뉴욕이면 뉴욕 재료, 시카고면 시카고 재료
  }
}
```

- 추상 팩토리에서 `CheesePizza`는 `ThinCrustDough`나 `MarinaraSauce`를 직접 알지 못함
    - `PizzaIngredientFactory`라는 인터페이스를 통해 재료를 받을 뿐
    - 뉴욕 팩토리를 주입하면 뉴욕 재료로
    - 시카로 팩토리를 주입하면 시칵고 재료로

→ *“관련 제품군의 **일관성이 타입 시스템에 의해 보장**”*

### 팩토리 메서드 vs 추상 팩토리 : 핵심 차이

두 패턴은 해결하는 문제의 차원이 다름…

***팩토리 메서드***

- **상속**을 통해 객체 생성을 위임
- ***“어떤 클래스를 인스턴스화할 것인가”***를 **서브클래스**가 결정
- 하나의 제품을 만드는 방법을 변형

***추상 팩토리***

- **구성**을 통해 객체 생성을 위임
- **관련 있는 제품군 전체를 만드는 팩토리 객체**를 주입받음
- 여러 제품이 일관되게 만들어져야 할 때 사용

> **팩토리 메서드**가 ***“이 자리에 어떤 제품을 넣을 것인가?”***의 문제라면,
**추상 팩토리**는 ***“이 제품군 전체를 어떤 변형으로 만들 것인가?”***의 문제
> 

### 의존성 역전의 원칙 (DIP)

> ***“추상화에 의존하되, 구상 클래스에 의존하지 말라”***
> 

***팩토리 패턴 적용 전***

- `PizzaStore`는 `NYStyleCheesePizza`, `ChicagoStylePepperoniPizza` 등 구체적인 클래스에 의존
- 고수준 모듈(`PizzaStore`)이 저수준 모듈(개별 피자 클래스)에 의존하는 구조

***팩토리 패턴 적용 후***

- `PizzaStore`는 `Pizza`라는 추상 타입에만 의존
- 개별 피자 클래스도 `Pizza` 추상 타입을 구현
- ***고수준과 저수준 모두 추상화에 의존*** → 의존의 방향이 뒤집힘

```
Before:
  PizzaStore → NYStyleCheesePizza
  PizzaStore → ChicagoStylePepperoniPizza
  PizzaStore → ...구체 클래스 직접 의존

After:
  PizzaStore → Pizza (추상)
  NYStyleCheesePizza → Pizza (추상)
  ChicagoStylePepperoniPizza → Pizza (추상)
  모두가 추상화에 의존
```

‘전략 패턴’과의 차이는,,

→ 전략 패턴은 행동(Strategy)을 인터페이스로 다뤘다면,

→ 팩터리 패턴은 객체 생성까지 인터페이스 뒤로 숨김

> 사용자 측이 구체적인 것을 알수록 결합도가 높아지고, 모를수록 결합도가 낮아짐
→ **팩토리 패턴**은 ***“어떤 구체 클래스의 인스턴스를 만드는가”***라는 지식을 사용하는 쪽에서 분리하는 것
> 

### 앞선 패턴들과의 관계

1. 전략 패턴
    - 행동을 캡슐화하고 교체 가능하게 만듬
    - “어떻게 할 것인가”의 유연성
2. 옵저버 패턴
    - 상태 변경의 전파를 구조화하고, 생산자와 소비자를 분리
    - “누구에게 알릴 것인가”의 유연성
3. 데코레이터 패턴
    - 기존 객체를 수정하지 않고 기능을 누적
    - “무엇을 더할 것인가”의 유연성
4. 팩토리 패턴
    - 객체 생성 자체를 캡슐화하고, 생성의 결정을 분리
    - “무엇을 만들 것인가”의 유연성

> **전략/옵저버/데코레이터**는 모두 객체를 사용하는 방식에 대한 패턴
**팩토리**는 여기서 한 단계 물러나, 객체를 만드는 방식에 대한 패턴

사용의 유연성이 아무리 높아도, 생성 시점에 구체 클래스에 결합되어 있으면 유연성은 떨어짐
→ 팩토리 패턴이 이 결합을 끊는다.
> 

### 더 넓은 맥락에서…

팩토리 패턴이 진짜 말하는 것은
→ “코드의 어느 부분이 구체적인 것을 알아야 하는가”라는 책임 분배의 문제

모든 코드가 구체적인 것을 알 필요가 없음.

→ 구체적인 것은 시스템의 극히 일부분(팩토리)에만 집중시키고

→ 나머지는 추상에만 의존하게 만드는 것

Q) 여기서 “이메일 대신 카카오톡으로 바꿔야 한다”는 요구사항이 생긴다면?

추가로 사용자 설정기반 알림수단 선택까지 반영된다면?

→ `if-else` ..

### 심플 팩토리 적용

```java
class NotificationFactory {
    static Notification create(User user, String title, String body) {
        switch (user.getPreference()) {
            case "email":
                return new EmailNotification(user.getEmail(), title, body);
            case "kakao":
                return new KakaoNotification(user.getPhone(), title, body);
            case "sms":
                return new SmsNotification(user.getPhone(), title, body);
            default:
                return new EmailNotification(user.getEmail(), title, body);
        }
    }
}

// 모든 서비스가 팩토리를 통해 생성
class OrderService {
    void completeOrder(Order order) {
        orderRepository.save(order);
        Notification notification = NotificationFactory.create(
            order.getUser(), "주문 완료", "주문이 완료되었습니다."
        );
        notification.send();
    }
}

class ShippingService {
    void shipOrder(Order order) {
        Notification notification = NotificationFactory.create(
            order.getUser(), "배송 시작", "상품이 발송되었습니다."
        );
        notification.send();
    }
}
```

→ 다른 알림을 추가하려면, `NotificationFactory`의 switch문에 case 하나만 추가하면 됨.

### 팩토리 메서드 적용 : 생성 정책의 다양성

심플 팩토리는 생성 로직이 한 곳에 모이지만, 로직이 복잡해지거나 여러 변형이 필요해지면 한계가 생김.

```java
// Creator — 데이터 접근 계층의 추상 클래스
abstract class DataAccessLayer {
    // 팩토리 메서드: 서브클래스가 결정
    abstract Connection createConnection();

    // 변하지 않는 흐름
    List<Record> query(String sql) {
        Connection conn = createConnection();  // 서브클래스에게 위임
        try {
            PreparedStatement stmt = conn.prepareStatement(sql);
            ResultSet rs = stmt.executeQuery();
            return mapResults(rs);
        } finally {
            conn.close();
        }
    }
}

// ConcreteCreator들 — 각각 다른 DB 커넥션을 생성
class PostgresDataAccess extends DataAccessLayer {
    Connection createConnection() {
```

### 프론트엔드에서 팩토리 패턴이 특별한 이유

프론트엔드에서 팩토리 패턴은 전통적인 추상 클래스 + 서브클래스 형태로 거의 등장하지 않음

JS/TS에서는 팩토리 함수, 설정 객체 기반 생성, Provider 패턴이라는 형태로 변형되어 쓰임

1. JS는 클래스보다 함수와 객체 리터럴이 더 자연스러운 언어
    - 추상 클래스를 상속해서 팩토리 메서드를 오버라이드하는 것보다,
    - 팩토리 함수를 만들어서 넘기는 것이 보통
2. 프론트엔드에서 “어떤 구체 클래스를 생성할 것인가”의 결정은 주로 런타임의 조건에 의해 이뤄짐
    - 컴파일 타임의 상속 계층보다 런타임의 함수 호출이 적합

### 팩토리 함수 - 가장 기본

```tsx
// API 에러 응답을 구체적인 에러 객체로 변환하는 팩토리
class NetworkError extends Error {
  constructor(message: string, public statusCode: number) {
    super(message);
    this.name = 'NetworkError';
  }
}

class AuthenticationError extends Error {
  constructor(message: string) {
    super(message);
    this.name = 'AuthenticationError';
  }
}

class ValidationError extends Error {
  constructor(message: string, public fields: Record<string, string>) {
    super(message);
    this.name = 'ValidationError';
  }
}

class NotFoundError extends Error {
  constructor(resource: string) {
    super(`${resource}를 찾을 수 없습니다`);
    this.name = 'NotFoundError';
  }
}

// 팩토리 함수: API 응답을 적절한 에러 타입으로 변환
function createApiError(response: Response, body?: any): Error {
  switch (response.status) {
    case 401:
    case 403:
      return new AuthenticationError(body?.message ?? '인증이 필요합니다');
    case 404:
      return new NotFoundError(body?.resource ?? '리소스');
    case 422:
      return new ValidationError(
        body?.message ?? '입력값이 올바르지 않습니다',
        body?.errors ?? {},
      );
    default:
      return new NetworkError(
        body?.message ?? '네트워크 오류가 발생했습니다',
        response.status,
      );
  }
}

// 사용하는 쪽은 구체적인 에러 생성 로직을 모름
async function fetchData(url: string) {
  const response = await fetch(url);
  if (!response.ok) {
    const body = await response.json().catch(() => null);
    throw createApiError(response, body);  // 팩토리에 위임
  }
  return response.json();
}

// 에러 처리도 타입별로 깔끔하게 분기
try {
  await fetchData('/api/users');
} catch (error) {
  if (error instanceof AuthenticationError) {
    redirect('/login');
  } else if (error instanceof ValidationError) {
    showFieldErrors(error.fields);
  } else if (error instanceof NotFoundError) {
    show404Page();
  }
}
```

- `createApiError()` 가 팩토리
- `fetchData()` 는 어떤 구체 에러 클래스가 존재하는지 알 필요 없고, `craeteApiError()` 만 호출
- 새로운 에러 타입이 추가되어도 팩토리만 수정

```tsx
  chart: ChartWidget,
  table: TableWidget,

  calendar: CalendarWidget,

  return <Component key={config.id} config={config} />;
}

// 새 위젯 등록은 레지스트리에 한 줄 추가
```

### 서비스/클라이언트 팩토리 - 환경에 따른 구현 교체

```tsx
// 스토리지 인터페이스
interface StorageClient {
  get(key: string): Promise<string | null>;
  set(key: string, value: string): Promise<void>;
  delete(key: string): Promise<void>;
}

// 프로덕션: 실제 서버 API 사용
class ApiStorageClient implements StorageClient {
  async get(key: string) {
    const res = await fetch(`/api/storage/${key}`);
    return res.ok ? res.text() : null;
  }
  async set(key: string, value: string) {
    await fetch(`/api/storage/${key}`, { method: 'PUT', body: value });
  }
  async delete(key: string) {
    await fetch(`/api/storage/${key}`, { method: 'DELETE' });
  }
}

// 개발: 로컬스토리지 사용 (서버 불필요)
class LocalStorageClient implements StorageClient {
  async get(key: string) { return localStorage.getItem(key); }
  async set(key: string, value: string) { localStorage.setItem(key, value); }
  async delete(key: string) { localStorage.removeItem(key); }
}

// 테스트: 인메모리 (부작용 없음)
class InMemoryStorageClient implements StorageClient {
  private store = new Map<string, string>();
  async get(key: string) { return this.store.get(key) ?? null; }
  async set(key: string, value: string) { this.store.set(key, value); }
  async delete(key: string) { this.store.delete(key); }
}

// 팩토리 함수: 환경에 따라 적절한 구현 생성
function createStorageClient(): StorageClient {
  if (process.env.NODE_ENV === 'test') {
    return new InMemoryStorageClient();
  }
  if (process.env.NODE_ENV === 'development') {
    return new LocalStorageClient();
  }
  return new ApiStorageClient();
}

// 사용하는 쪽은 어떤 구현인지 전혀 모름
const storage = createStorageClient();
await storage.set('user-preference', JSON.stringify(prefs));
```

### HTTP 클라이언트 팩토리 - 설정 기반 생성

Axios의 `axios.create()`는 교과서적인 팩토리 메서드

```tsx
// 기본 설정이 다른 여러 HTTP 클라이언트를 팩토리로 생성
function createApiClient(baseURL: string, options?: {
  withAuth?: boolean;
  timeout?: number;
  retryCount?: number;
}) {
  const client = axios.create({
    baseURL,
    timeout: options?.timeout ?? 10000,
    headers: { 'Content-Type': 'application/json' },
  });

  // 조건에 따라 인터셉터(데코레이터)를 선택적으로 추가
  if (options?.withAuth) {
    client.interceptors.request.use((config) => {
      config.headers.Authorization = `Bearer ${getToken()}`;
      return config;
    });
  }

  if (options?.retryCount) {
    // 재시도 로직 추가
    client.interceptors.response.use(undefined, retryInterceptor(options.retryCount));
  }

  return client;
}

// 서비스별로 다른 설정의 클라이언트 생성
const userApi = createApiClient('/api/users', { withAuth: true });
const publicApi = createApiClient('/api/public', { timeout: 5000 });
const paymentApi = createApiClient('/api/payments', {
  withAuth: true,
  retryCount: 3,
  timeout: 30000,
});

// 사용하는 쪽은 인증, 재시도, 타임아웃 설정을 의식하지 않음
const users = await userApi.get('/');
const products = await publicApi.get('/products');
```

- 팩토리 패턴과 데코레이터 패턴이 자연스럽게 결합
- 팩토리가 객체를 생성하고, 조건에 따라 데코레이터(인터셉터)를 선택적으로 감쌈

### 추상 팩토리의 프론트엔드 변형 - 테마/디자인 시스템

```tsx
// 추상 팩토리: 관련 컴포넌트군의 생성 인터페이스
interface UIKitFactory {
  Button: React.ComponentType<ButtonProps>;
  Input: React.ComponentType<InputProps>;
  Card: React.ComponentType<CardProps>;
  Badge: React.ComponentType<BadgeProps>;
}

// Concrete Factory 1: 기본 UI 킷
const defaultUIKit: UIKitFactory = {
  Button: DefaultButton,
  Input: DefaultInput,
  Card: DefaultCard,
  Badge: DefaultBadge,
};

// Concrete Factory 2: 컴팩트 UI 킷 (밀도 높은 대시보드용)
const compactUIKit: UIKitFactory = {
  Button: CompactButton,
  Input: CompactInput,
  Card: CompactCard,
  Badge: CompactBadge,
};

// Concrete Factory 3: 고대비 접근성 UI 킷
const highContrastUIKit: UIKitFactory = {
  Button: HighContrastButton,
  Input: HighContrastInput,
  Card: HighContrastCard,
  Badge: HighContrastBadge,
};

// React Context로 팩토리 주입
const UIKitContext = createContext<UIKitFactory>(defaultUIKit);

function UIKitProvider({ variant, children }: {
  variant: 'default' | 'compact' | 'high-contrast';
  children: React.ReactNode;
}) {
  const kit = {
    default: defaultUIKit,
    compact: compactUIKit,
    'high-contrast': highContrastUIKit,
  }[variant];

  return <UIKitContext.Provider value={kit}>{children}</UIKitContext.Provider>;
}

// 사용하는 쪽은 어떤 UI 킷인지 모름
// 자동으로 일관된 컴포넌트군이 사용됨
function UserForm() {
  const { Button, Input, Card } = useContext(UIKitContext);

  return (
    <Card>
      <Input label="이름" />
      <Input label="이메일" type="email" />
      <Button variant="primary">저장</Button>
    </Card>
  );
}

// 앱 레벨에서 팩토리 교체 → 모든 하위 컴포넌트에 일관 적용
<UIKitProvider variant="compact">
  <Dashboard />  {/* 안의 모든 컴포넌트가 compact 버전 */}
</UIKitProvider>
```
