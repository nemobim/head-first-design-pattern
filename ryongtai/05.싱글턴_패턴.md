# 싱글턴 패턴

## 싱글턴 패턴의 근본적 문제의식과 핵심 철학

### 1. 실글턴 패턴이 해결하려는 근본 문제: “유일성의 보장”

앞선 패턴들은 ‘유연성’에 관한 패턴 
→ 행동을 교체하고, 변경을 전파하고, 기능을 누적하고, 생성을 위임

> 싱글턴 패턴은 ***‘유연성’***이 아니라 ***‘제약’***에 관한 패턴
> 

소프트웨어에서 특정 객체가 시스템 전체에서 단 하나만 존재해야 하는 상황이 있음:
→ 데이터베이스 커넥션 풀 / 로그 관리자 / 설정 객체 / 캐시 관리자 등..
→ 이런 객체가 여러 개 생기면, 리소스 낭비 / 설정 객체 중복 / 데이터 불일치 등의 문제 발생

> 싱글턴 패턴은 바로 이.. 
”***특정 클래스의 인스턴스가 하나만 존재하도록 보장하고, 그 인스턴스에 대한 전역 접근점을 제공”***
> 

### 2. 왜 전역 변수로는 부족할까?

‘하나만 있어야 하는 객체’라면 전역 변수를 사용할 수도 있지 않을까?

```tsx
// 전역 변수로 관리
const globalConfig = {
	apiUrl: "https://api.example.com",
	timeout: 5000,
	debug: false,
}

globalConfig.timeout = 10000;
```

이런 전역 변수에는 두 가지 근본적인 한계가 있음.

1. “하나만 존재한다”는 것을 강제할 수 없음.
    - 전역 변수의 관리는 Convention에 의존
    - 누군가의 실수를 구조적으로 막지 못함
2. 초기화 시점을 제어할 수 없음.
    - 전역 변수는 프로그램이 시작될 때 생성
    - 하지만 DB Connection처럼 무거운 리소스는 실제로 필요할 때 생성해야 하는 경우가 있음
    → 사용되지 않을 수도 있는 리소스를 무조건 생성하는 것은 리소스 낭비

> 싱글턴 패턴은
→ 구조적으로 인스턴스가 하나만 존재하도록 강제
→ **지연 초기화**를 통해 필요한 시점에 생성
> 

### 3. 싱글턴 패턴은 구조적 해결

전통적 싱글턴의 구조는 단순

```tsx
class Singeton {
  // 유일한 인스턴스를 저장하는 정적 변수
  private static instance: Singleton | null = null;

  // 생성자를 private으로 막아서 외부에서 new를 사용할 수 없게 함
  private constructor() {
    // 초기화 로직
  }

  // 유일한 접근점
  static getInstance(): Singleton {
    if (!Singleton.instance) {
      Singleton.instance = new Singleton();
    }
    return Singleton.instance;
  }
}

// new Singleton()  → 컴파일 에러! private constructor
// Singleton.getInstance() → 항상 같은 인스턴스 반환
const a = Singleton.getInstance();
const b = Singleton.getInstance();
console.log(a === b);  // true — 동일한 인스턴스
```

- `private constructor`로 외부 생성을 차단
- 정적 변수로 유일한 인스턴스를 보관
- 정적 메서드로 유일한 접근점 제공
- `getInstance()`가 호출될 때만 인스턴스가 생성되고(지언 초기화) 이후에는 같은 인스턴스 반환

### 4. 멀티스레딩 문제: 싱글턴의 함정

두 스레드가 동시에 `getInstance()`를 호출한다면…?

```tsx
Thread A: if (!instance)    → null이네, 생성해야지
Thread B: if (!instance)    → null이네, 생성해야지 (A가 아직 할당 전!)
Thread A: instance = new Singleton()
Thread B: instance = new Singleton()  → 인스턴스가 두 개!
```

두 스레드가 `instance`가 `null` 인 것을 동시에 확인하고, 각각 인스턴스를 생성
→ “유일성 보장”이라는 싱글턴의 존재 이유가 무너지는 순간!!!

책에서 제시하는 해결 방법 3가지

1. 동기화 (synchronized) : `getInstance()`에 락을 걸어서 한 번에 하나의 스레드만 접근하게 함.
    - 안전하지만, 실제 동기화가 필요한 것은 최초 생성 시뿐인데 매번 락을 거는 비용이 발생
2. 이른 초기화 (eager initailization) : 클래스 로딩 시점에 인스턴스를 미리 만들어 둠
    - 스레드 안전, 지연 초기화의 이점을 포기
3. DCL (Double-Checked Locking) : 인스턴스가 없을 때만 동기화 블록에 진입하고, 블록 안에서 한 번 더 확인
    - 성능과 언전성 모두 해결, 코드가 복잡..

### 5. 싱글턴 패턴의 논쟁 : 왜 “안티패턴”이라고 불리는가?

논란이 많은 싱글턴 패턴…..

1. 싱글턴은 사실상 전역 상태
    - `Singleton.getInstance()` 는 어디서든 접근 가능
    - 즉, “누가 언제 상태를 바꿨는지 추적하기 어렵다”
    - 전역 변수를 객체지향적으로 포장한 것이라는 비판…
2. 테스트를 어렵게 만듦
    - 싱글턴은 인스턴스가 하나뿐이므로, 테스트 간에 상태가 공유
    - Mock으로 교체하기 어려움..
3. 단일 책임 원칙(SRP)을 위반
    - 싱글턴 클래스는 자신의 본래 책임 외에 “자신의 인스턴스가 하나만 존재하도록 관리하는” 책임을 갖음
4. 의존성을 숨김
    - 함수가 내부에서 `Singleton.getInstance()`를 호출하면, 그 함수의 시그니처만으로는 싱글턴에 의존한다는 사실을 알 수 없음
    - 의존성이 명시적이지 않고 숨겨져 있음
    
    ```tsx
    // 의존성이 숨겨져 있음 — 시그니처만 보면 알 수 없음
    function processOrder(order: Order): void {
      // 내부에서 싱글턴에 의존
      const config = AppConfig.getInstance();
      const logger = Logger.getInstance();
      const cache = CacheManager.getInstance();
      // ...
    }
    
    // 의존성이 명시적 — 시그니처에 드러남
    function processOrder(
      order: Order,
      config: AppConfig,
      logger: Logger,
      cache: CacheManager,
    ): void {
      // ...
    }
    ```
    

### 6. 그럼에도.. 싱글턴이 필요한 경우는..?

> 핵심은 언제 적절한가를 판단하는 것!
> 

싱글턴이 적절한 경우는 두 가지 조건이 동시에 충족될 때:

1. 인스턴스가 하나만 존재해야 하는 것이 비즈니스의 요구사항일 때
2. 전역 접근이 필요할 때

- 대표적 예시 - DB Connection Pool :
    - 커넥션 풀이 여러 개 생기면 커넥션이 낭비되고 관리가 어려워짐
    - 하드웨어와 상호작용하는 드라이버도 마찬가지
    
    → 하지만, *현대 소프트웨어 개발에서는 “유일성 보장”이 필요하더라도, 전통적인 싱글턴 패턴 대신 DI 컨테이너의 스코프 관리로 해결하는 것이 주류* (Spring의 빈 스코프)
    
    → 객체 자체가 자신의 유일성을 관리하는 것이 아니라, 외부(컨테이너)가 유일성을 보장하는 구조
    → 이렇게 하면 싱글턴의 장점(유일한 인스턴스)은 유지되면서 단점(테스트, 숨겨진 의존)을 피할 수 있음
    

### 7. 앞선 패턴들과의 관계

> 전략, 옵저버, 데코레이터, 팩토리는 모두 결합도를 낮추고 유연성을 높이는 방향의 패턴
→ “구현이 아닌 인터페이스에 의존하라”, “변하는 것을 캡슐화하라”
> 

싱글은 제약을 부과하는 패턴! 유연성이 아니라 통제를 추구
→ *“이 인스턴스는 하나만 존재해야 한다.”
→ 덕분에 다른 패턴들이 만들어낸 유연성을 싱글턴이 경직시킬 수 있음..*

### 8. 더 넓은 맥락에서의 싱글턴 패턴

*“인스턴스 하나만 만드는 트릭”* → 본질을 놓치는 접근
→ *싱글턴이 진짜 다루는 것은 **“공유 자원의 관리”***

- **전통적인 싱글턴**은 ***“자원 스스로가 자신의 유일성을 관리한다”***
- **현대에서는 DI**, ***“외부 컨테이너가 자원의 수명과 유일성을 관리한다”***
- **모듈 시스템**은 ***“모듈의 캐싱 매커니즘이 자연스럽게 유일성을 보장한다”***

프론트엔드에서 싱글턴이 중요한 이유:
→ 브라우저 환경 자체가 싱글턴적 자원으로 가득하다…!

- `window`, `document`, `localStorage`, `navigator` 는 모두 유일한 전역 객체
- React의 루트 스토어, Next.js의 서버사이드 캐시, WebSocket 연결도 보통 하나만 존재..

문제는 어떤 방식으로 싱글턴을 만드냐는 것:
→ 전통적인 방식 대신, **모듈 캐싱, Context, 클로저 같은 JS 고유 매커니즘**이 그 역할을 대신..

## 실전 예시 - 전통적인 프로그래밍 관점에서의 싱글턴 패턴

### 1. 싱글턴이 실무에서 등장하는 전형적인 장면

싱글턴이 해결하는 문제:

- 비용이 큰 공유 자원을 한 번만 초기화해야 함
- 어디서든 동일한 인스턴스에 접근해야 함

DB 커넥션 풀:

```java
// ⚠️ 싱글턴이 없을 때: 매 요청마다 커넥션 생성
class UserService {
    void getUser(String id) {
        // 💣 매번 새 커넥션 생성 — 비용이 크다
        Connection conn = DriverManager.getConnection("jdbc:postgresql://...");
        // 쿼리 실행
        conn.close();
    }
}

class OrderService {
    void getOrder(String id) {
        // 💣 여기서도 또 새 커넥션 생성
        Connection conn = DriverManager.getConnection("jdbc:postgresql://...");
        // 쿼리 실행
        conn.close();
    }
}
// -> 동시 요청 1000개 → 커넥션 1000개 → 데이터베이스 과부하

// ✅ 싱글턴 커넥션 풀로 해결
class DatabasePool {
    private static DatabasePool instance;
    private HikariDataSource dataSource;

    private DatabasePool() {
        HikariConfig config = new HikariConfig();
        config.setJdbcUrl("jdbc:postgresql://...");
        config.setMaximumPoolSize(20);  // 최대 20개 커넥션만 유지
        this.dataSource = new HikariDataSource(config);
    }

    static synchronized DatabasePool getInstance() {
        if (instance == null) {
            instance = new DatabasePool();
        }
        return instance;
    }

    Connection getConnection() throws SQLException {
        return dataSource.getConnection();
    }
}

// 어디서든 동일한 풀에서 커넥션을 빌림
class UserService {
    void getUser(String id) {
        Connection conn = DatabasePool.getInstance().getConnection();
        // 쿼리 실행 후 반환 (풀로 돌아감)
    }
}
```

설정 관리:

```java
class AppConfig {
    private static AppConfig instance;
    private final Properties properties;

    private AppConfig() {
        this.properties = new Properties();
        // 파일에서 설정 로드 — 비용이 있는 I/O 작업
        try (InputStream is = getClass().getResourceAsStream("/application.properties")) {
            properties.load(is);
        }
    }

    static AppConfig getInstance() {
        if (instance == null) {
            instance = new AppConfig();
        }
        return instance;
    }

    String get(String key) {
        return properties.getProperty(key);
    }

    int getInt(String key, int defaultValue) {
        String value = properties.getProperty(key);
        return value != null ? Integer.parseInt(value) : defaultValue;
    }
}

// 시스템 어디서든 동일한 설정에 접근
String apiUrl = AppConfig.getInstance().get("api.url");
int timeout = AppConfig.getInstance().getInt("api.timeout", 5000);
```

### 2. 현대적 대안 : DI 컨테이너에 의한 싱글턴 스코프

```java
// Spring — 기본 빈 스코프가 싱글턴
@Service  // Spring 컨테이너가 인스턴스를 하나만 생성해서 관리
class UserService {
    private final DatabasePool pool;

    // 생성자 주입 — 의존성이 명시적
    UserService(DatabasePool pool) {
        this.pool = pool;
    }
}

// DatabasePool도 Spring이 관리
@Configuration
class DatabaseConfig {
    @Bean  // 이 메서드의 반환값은 하나만 생성되어 공유됨
    DatabasePool databasePool() {
        return new DatabasePool("jdbc:postgresql://...");
    }
}
```

- `DatabasePool`은 여전히 시스템에 하나만 존재
- 하지만, `DatabasePool` 자체가 자신의 유일성을 관리하지 않음
- Spring 컨테이너가 외부에서 유일성을 보장
- `DatabasePool`의 생성자는 `public`이므로 테스트에서 자유롭게 인스턴스를 만들 수 있고, Mock으로 교체할 수도 있음

```java
// 테스트: 싱글턴이 아닌 별도 인스턴스를 자유롭게 생성
@Test
void testUserService() {
    DatabasePool mockPool = mock(DatabasePool.class);
    UserService service = new UserService(mockPool);
    // 테스트 간 상태 공유 없음, 완전히 독립적
}
```

→ ***“싱글턴의 이점은 취하되 싱글턴 패턴의 구현은 쓰지 않는..” 현대적인 접근***
→ 객체의 유일성은 필요하지만, 스스로 관리하는 것이 아니라 외부 시스템(DI 컨테이너)이 관리!

### 3. NestJs - DI 기반 싱글턴

```tsx
// NestJS에서 @Injectable()은 기본적으로 싱글턴 스코프
@Injectable()
class CacheService {
  private store = new Map<string, { value: any; expiresAt: number }>();

  get(key: string): any | null {
    const entry = this.store.get(key);
    if (!entry || entry.expiresAt < Date.now()) {
      this.store.delete(key);
      return null;
    }
    return entry.value;
  }

  set(key: string, value: any, ttlMs: number) {
    this.store.set(key, { value, expiresAt: Date.now() + ttlMs });
  }
}

// 어떤 서비스에서 주입받든 같은 인스턴스
@Injectable()
class UserService {
  constructor(private cache: CacheService) {}  // 의존성 명시적
}

@Injectable()
class OrderService {
  constructor(private cache: CacheService) {}  // 같은 CacheService 인스턴스
}
```

- NestJS의 `@Injectable()`은 기본 스코프가 싱글턴
- 하지만, 클래스 자체에 `private constructor`나 `getInstance()`가 없음
- 유일성은 NestJS의 IoC 컨테이너가 관리!

## 실무 예시 - 프론트엔드 생태계 관점에서 싱글턴

### 1. 프론트엔드의 싱글턴

FE에서 전통적 패턴은 거의 쓰이지 않음.
→ JS/TS에 이미 싱글턴을 자연스럽게 만드는 매커니즘 존재! - **모듈 시스템**

모듈 시스템:

- ES모듈에서 `import`는 모듈을 한 번만 실행하고 그 결과를 캐싱
- 캐싱된 exports를 반환
- 이것은 모듈 수준에서 내보내는 객체가 자동으로 싱글턴이 된다는 뜻,,

```tsx
// db.ts
// 이 모듈이 처음 import될 때 한 번만 실행됨
console.log("DB 모듈 초기화");  // 한 번만 출력됨

class DatabaseClient {
  constructor(private url: string) {
    console.log("DB 커넥션 생성");  // 한 번만 출력됨
  }
  query(sql: string) { /* ... */ }
}

// 모듈 스코프에서 인스턴스 생성 → 자동으로 싱글턴
export const db = new DatabaseClient(process.env.DATABASE_URL!);
```

```tsx
// user-service.ts
import { db } from './db';  // 캐싱된 인스턴스
db.query("SELECT * FROM users");

// order-service.ts
import { db } from './db';  // 같은 인스턴스!
db.query("SELECT * FROM orders");

// 두 파일의 db는 === 으로 같다
```

### 2. 모듈 레벨 싱글턴 - API 클라이언트

```tsx
// api-client.ts — 모듈이 곧 싱글턴
import axios from 'axios';

// 이 모듈이 import되는 순간 한 번만 실행
const apiClient = axios.create({
  baseURL: process.env.NEXT_PUBLIC_API_URL,
  timeout: 10000,
  headers: { 'Content-Type': 'application/json' },
});

// 인터셉터 설정도 한 번만 실행
apiClient.interceptors.request.use((config) => {
  const token = typeof window !== 'undefined'
    ? localStorage.getItem('token')
    : null;
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

apiClient.interceptors.response.use(
  (response) => response,
  async (error) => {
    if (error.response?.status === 401) {
      // 토큰 갱신 로직
    }
    return Promise.reject(error);
  },
);

// 이것 하나를 모든 곳에서 import
export default apiClient;
```

### 3. Prisma Client - Next.js에서..

개발 환경에서 HMR이 발생할 때마다 모듈이 다시 실행되면서 Prisma Client 인스턴스가 중복 생성됨.

```tsx
// ❌ 개발 환경에서 HMR마다 새 인스턴스 생성 → 커넥션 고갈
// lib/prisma.ts
import { PrismaClient } from '@prisma/client';
export const prisma = new PrismaClient();
// HMR 발생 → 모듈 재실행 → new PrismaClient() 또 호출
// 이전 인스턴스의 커넥션은 닫히지 않고 남아있음
// 결과: "Too many database connections" 에러 
```

```tsx
// ✅ globalThis를 활용한 싱글턴 보장
// lib/prisma.ts
import { PrismaClient } from '@prisma/client';

function createPrismaClient(): PrismaClient {
  return new PrismaClient({
    log: process.env.NODE_ENV === 'development' ? ['query'] : [],
  });
}

// globalThis는 HMR에도 유지되는 전역 객체
const globalForPrisma = globalThis as unknown as {
  prisma: PrismaClient | undefined;
};

// 이미 존재하면 재사용, 없으면 생성
export const prisma = globalForPrisma.prisma ?? createPrismaClient();

// 개발 환경에서만 globalThis에 저장 (프로덕션에서는 불필요)
if (process.env.NODE_ENV !== 'production') {
  globalForPrisma.prisma = prisma;
}
```

개발 서버의 HMR이 모듈 캐시를 무효화하기 때문에, ES 모듈 캐싱만으로는 부족..
→ HMR에도 영향받지 않는 전역 객체(`globalThis`)를 사용해서 싱글턴을 보장해야함

### 4. Zustand/Redux Store - 상태 관리의 싱글턴

```tsx
// store.ts — 모듈 레벨 싱글턴
import { create } from 'zustand';
import { devtools, persist } from 'zustand/middleware';

// create()의 반환값이 모듈 스코프에서 한 번만 생성됨
export const useAuthStore = create<AuthState>()(
  devtools(
    persist(
      (set, get) => ({
        user: null,
        token: null,

        login: async (credentials) => {
          const response = await api.post('/auth/login', credentials);
          set({
            user: response.data.user,
            token: response.data.token,
          });
        },

        logout: () => {
          set({ user: null, token: null });
        },

        isAuthenticated: () => get().token !== null,
      }),
      { name: 'auth-storage' },
    ),
  ),
);

// 어떤 컴포넌트에서 import하든 같은 스토어
// Header.tsx
const user = useAuthStore(state => state.user);

// Sidebar.tsx
const user = useAuthStore(state => state.user);  // 같은 상태

// LoginForm.tsx
const login = useAuthStore(state => state.login);
login(credentials);  // → Header, Sidebar 모두 자동 업데이트
```

스토어가 싱글턴이 아니면 스토어의 옵저버 패턴이 무의미해짐
→ 하나의 스토어 상태가 바뀌어야 그 스토어를 구독하는 모든 컴포넌트가 업데이트 됨.
→ 두 개면,, 한 쪽의 변경이 다른 쪽의 구독자에게 전파되지 않음

### 5. 이벤트 버스 - 싱글턴

이벤트 기반 통신에서 이벤트 버스는 반드시 싱글턴이어야 함!
→ 버스가 두 개면 한 쪽에서 발행한 이벤트를 다른 쪽의 구독자가 받지 못함.

```tsx
// event-bus.ts — 모듈 레벨 싱글턴
type Handler = (...args: any[]) => void;

class EventBus {
  private listeners = new Map<string, Set<Handler>>();

  on(event: string, handler: Handler): () => void {
    if (!this.listeners.has(event)) {
      this.listeners.set(event, new Set());
    }
    this.listeners.get(event)!.add(handler);
    return () => this.listeners.get(event)?.delete(handler);
  }

  emit(event: string, ...args: any[]) {
    this.listeners.get(event)?.forEach(handler => handler(...args));
  }

  clear() {
    this.listeners.clear();
  }
}

// 모듈 스코프에서 하나만 생성 → 앱 전체에서 공유
export const eventBus = new EventBus();
```

```tsx

}, []);

// 컴포넌트 B에서 발행
```

### 6. WebSocket 매니저 - 커넥션 유일성

WebSocket 연결은 싱글턴의 고전적인 정당화 사례..
→ 같은 서버에 대한 연결이 여러 개 생기면 서버 리소스 낭비이고, 메시지 순서 보장도 어려워짐

```tsx
// websocket-manager.ts
class WebSocketManager {
  private ws: WebSocket | null = null;
  private listeners = new Map<string, Set<Function>>();
  private reconnectAttempts = 0;
  private maxReconnectAttempts = 5;

  connect(url: string) {
    if (this.ws?.readyState === WebSocket.OPEN) return;

    this.ws = new WebSocket(url);

    this.ws.onmessage = (event) => {
      const { type, payload } = JSON.parse(event.data);
      this.listeners.get(type)?.forEach(cb => cb(payload));
    };

    this.ws.onclose = () => {
      if (this.reconnectAttempts < this.maxReconnectAttempts) {
        this.reconnectAttempts++;
        setTimeout(() => this.connect(url), 2 ** this.reconnectAttempts * 1000);
      }
    };

    this.ws.onopen = () => {
      this.reconnectAttempts = 0;
    };
  }

  subscribe(type: string, callback: Function): () => void {
    if (!this.listeners.has(type)) {
      this.listeners.set(type, new Set());
    }
    this.listeners.get(type)!.add(callback);
    return () => this.listeners.get(type)?.delete(callback);
  }

  send(type: string, payload: unknown) {
    if (this.ws?.readyState === WebSocket.OPEN) {
      this.ws.send(JSON.stringify({ type, payload }));
    }
  }

  disconnect() {
    this.ws?.close();
    this.ws = null;
  }
}

// 모듈 레벨 싱글턴
export const wsManager = new WebSocketManager();
```

### 7. 클로저 기반 싱글턴 - `private constructor`의 JS적 대안

클래스가 아닌 함수와 클로저로도 싱글턴을 구현할 수 있음.

```tsx
// 클로저로 인스턴스를 캡슐화
function createLogger() {
  let instance: Logger | null = null;

  return {
    getInstance(): Logger {
      if (!instance) {
        instance = {
          logs: [] as string[],
          info(message: string) {
            const entry = `[INFO ${new Date().toISOString()}] ${message}`;
            this.logs.push(entry);
            console.log(entry);
          },
          error(message: string) {
            const entry = `[ERROR ${new Date().toISOString()}] ${message}`;
            this.logs.push(entry);
            console.error(entry);
          },
          getLogs() {
            return [...this.logs];
          },
        };
      }
      return instance;
    },
  };
}

// IIFE로 즉시 실행하여 팩토리를 닫음
export const Logger = createLogger();

// 사용
const logger = Logger.getInstance();
logger.info("앱 시작");

const sameLogger = Logger.getInstance();
console.log(logger === sameLogger);  // true
```

### 8. React Context - 컴포넌트 트리 스코프의 싱글턴

컴포넌트 트리 내에서의 싱글턴을 제공
→ 전역 싱글턴보다 스코프가 제한적, 싱글턴의 단점(전역 오염)을 줄이고 이점(공유 인스턴스)은 유지

```tsx
// 테마 컨텍스트 — 트리 내에서 하나만 존재
const ThemeContext = createContext<ThemeContextType | null>(null);

function ThemeProvider({ children }: { children: React.ReactNode }) {
  const [theme, setTheme] = useState<'light' | 'dark'>('light');

  // 이 value 객체가 트리 내 모든 소비자에게 공유됨
  const value = useMemo(() => ({
    theme,
    toggle: () => setTheme(prev => prev === 'light' ? 'dark' : 'light'),
  }), [theme]);

  return (
    <ThemeContext.Provider value={value}>
      {children}
    </ThemeContext.Provider>
  );
}

function useTheme() {
  const context = useContext(ThemeContext);
  if (!context) throw new Error('useTheme must be used within ThemeProvider');
  return context;
}

// 트리 내 어디서든 같은 테마에 접근
function Header() {
  const { theme, toggle } = useTheme();
  return <button onClick={toggle}>{theme}</button>;
}

function Content() {
  const { theme } = useTheme();  // Header와 같은 context 값
  return <div className={theme}>...</div>;
}
```

```tsx
// 전역 싱글턴이었다면 불가능한 구조
<ThemeProvider initialTheme="light">
  <MainApp />      {/* 라이트 테마 */}
</ThemeProvider>

<ThemeProvider initialTheme="dark">
  <AdminPanel />   {/* 다크 테마 — 독립적 */}
</ThemeProvider>
```
