# 싱글턴 패턴

## 싱글턴 패턴의 근본적 문제의식과 핵심 철학

### 1. 실글턴 패턴이 해결하려는 근본 문제: “유일성의 보장”

앞선 패턴들은 ‘유연성’에 관한 패턴 
→ 행동을 교체하고, 변경을 전파하고, 기능을 누적하고, 생성을 위임

> 싱글턴 패턴은 ***‘유연성’***이 아니라 ***‘제약’***에 관한 패턴
> 

소프트웨어에서 특정 객체가 시스템 전체에서 단 하나만 존재해야 하는 상황이 있음:
→ 데이터베이스 커넥션 풀 / 로그 관리자 / 설정 객체 / 캐시 관리자 등..
→ 이런 객체가 여러 개 생기면, 리소스 낭비 / 설정 객체 중복 / 데이터 불일치 등의 문제 발생

> 싱글턴 패턴은 바로 이.. 
”***특정 클래스의 인스턴스가 하나만 존재하도록 보장하고, 그 인스턴스에 대한 전역 접근점을 제공”***
> 

### 2. 왜 전역 변수로는 부족할까?

‘하나만 있어야 하는 객체’라면 전역 변수를 사용할 수도 있지 않을까?

```tsx
// 전역 변수로 관리
const globalConfig = {
	apiUrl: "https://api.example.com",
	timeout: 5000,
	debug: false,
}

globalConfig.timeout = 10000;
```

이런 전역 변수에는 두 가지 근본적인 한계가 있음.

1. “하나만 존재한다”는 것을 강제할 수 없음.
    - 전역 변수의 관리는 Convention에 의존
    - 누군가의 실수를 구조적으로 막지 못함
2. 초기화 시점을 제어할 수 없음.
    - 전역 변수는 프로그램이 시작될 때 생성
    - 하지만 DB Connection처럼 무거운 리소스는 실제로 필요할 때 생성해야 하는 경우가 있음
    → 사용되지 않을 수도 있는 리소스를 무조건 생성하는 것은 리소스 낭비

> 싱글턴 패턴은
→ 구조적으로 인스턴스가 하나만 존재하도록 강제
→ **지연 초기화**를 통해 필요한 시점에 생성
> 

### 3. 싱글턴 패턴은 구조적 해결

전통적 싱글턴의 구조는 단순

```tsx
class Singeton {
  // 유일한 인스턴스를 저장하는 정적 변수
  private static instance: Singleton | null = null;

  // 생성자를 private으로 막아서 외부에서 new를 사용할 수 없게 함
  private constructor() {
    // 초기화 로직
  }

  // 유일한 접근점
  static getInstance(): Singleton {
    if (!Singleton.instance) {
      Singleton.instance = new Singleton();
    }
    return Singleton.instance;
  }
}

// new Singleton()  → 컴파일 에러! private constructor
// Singleton.getInstance() → 항상 같은 인스턴스 반환
const a = Singleton.getInstance();
const b = Singleton.getInstance();
console.log(a === b);  // true — 동일한 인스턴스
```

- `private constructor`로 외부 생성을 차단
- 정적 변수로 유일한 인스턴스를 보관
- 정적 메서드로 유일한 접근점 제공
- `getInstance()`가 호출될 때만 인스턴스가 생성되고(지언 초기화) 이후에는 같은 인스턴스 반환

### 4. 멀티스레딩 문제: 싱글턴의 함정

두 스레드가 동시에 `getInstance()`를 호출한다면…?

```tsx
Thread A: if (!instance)    → null이네, 생성해야지
Thread B: if (!instance)    → null이네, 생성해야지 (A가 아직 할당 전!)
Thread A: instance = new Singleton()
Thread B: instance = new Singleton()  → 인스턴스가 두 개!
```

두 스레드가 `instance`가 `null` 인 것을 동시에 확인하고, 각각 인스턴스를 생성
→ “유일성 보장”이라는 싱글턴의 존재 이유가 무너지는 순간!!!

책에서 제시하는 해결 방법 3가지

1. 동기화 (synchronized) : `getInstance()`에 락을 걸어서 한 번에 하나의 스레드만 접근하게 함.
    - 안전하지만, 실제 동기화가 필요한 것은 최초 생성 시뿐인데 매번 락을 거는 비용이 발생
2. 이른 초기화 (eager initailization) : 클래스 로딩 시점에 인스턴스를 미리 만들어 둠
    - 스레드 안전, 지연 초기화의 이점을 포기
3. DCL (Double-Checked Locking) : 인스턴스가 없을 때만 동기화 블록에 진입하고, 블록 안에서 한 번 더 확인
    - 성능과 언전성 모두 해결, 코드가 복잡..

### 5. 싱글턴 패턴의 논쟁 : 왜 “안티패턴”이라고 불리는가?

논란이 많은 싱글턴 패턴…..

1. 싱글턴은 사실상 전역 상태
    - `Singleton.getInstance()` 는 어디서든 접근 가능
    - 즉, “누가 언제 상태를 바꿨는지 추적하기 어렵다”
    - 전역 변수를 객체지향적으로 포장한 것이라는 비판…
2. 테스트를 어렵게 만듦
    - 싱글턴은 인스턴스가 하나뿐이므로, 테스트 간에 상태가 공유
    - Mock으로 교체하기 어려움..
3. 단일 책임 원칙(SRP)을 위반
    - 싱글턴 클래스는 자신의 본래 책임 외에 “자신의 인스턴스가 하나만 존재하도록 관리하는” 책임을 갖음
4. 의존성을 숨김
    - 함수가 내부에서 `Singleton.getInstance()`를 호출하면, 그 함수의 시그니처만으로는 싱글턴에 의존한다는 사실을 알 수 없음
    - 의존성이 명시적이지 않고 숨겨져 있음
    
    ```tsx
    // 의존성이 숨겨져 있음 — 시그니처만 보면 알 수 없음
    function processOrder(order: Order): void {
      // 내부에서 싱글턴에 의존
      const config = AppConfig.getInstance();
      const logger = Logger.getInstance();
      const cache = CacheManager.getInstance();
      // ...
    }
    
    // 의존성이 명시적 — 시그니처에 드러남
    function processOrder(
      order: Order,
      config: AppConfig,
      logger: Logger,
      cache: CacheManager,
    ): void {
      // ...
    }
    ```
    

### 6. 그럼에도.. 싱글턴이 필요한 경우는..?

> 핵심은 언제 적절한가를 판단하는 것!
> 

싱글턴이 적절한 경우는 두 가지 조건이 동시에 충족될 때:

1. 인스턴스가 하나만 존재해야 하는 것이 비즈니스의 요구사항일 때
2. 전역 접근이 필요할 때

- 대표적 예시 - DB Connection Pool :
    - 커넥션 풀이 여러 개 생기면 커넥션이 낭비되고 관리가 어려워짐
    - 하드웨어와 상호작용하는 드라이버도 마찬가지
    
    → 하지만, *현대 소프트웨어 개발에서는 “유일성 보장”이 필요하더라도, 전통적인 싱글턴 패턴 대신 DI 컨테이너의 스코프 관리로 해결하는 것이 주류* (Spring의 빈 스코프)
    
    → 객체 자체가 자신의 유일성을 관리하는 것이 아니라, 외부(컨테이너)가 유일성을 보장하는 구조
    → 이렇게 하면 싱글턴의 장점(유일한 인스턴스)은 유지되면서 단점(테스트, 숨겨진 의존)을 피할 수 있음
    

### 7. 앞선 패턴들과의 관계

> 전략, 옵저버, 데코레이터, 팩토리는 모두 결합도를 낮추고 유연성을 높이는 방향의 패턴
→ “구현이 아닌 인터페이스에 의존하라”, “변하는 것을 캡슐화하라”
> 

싱글은 제약을 부과하는 패턴! 유연성이 아니라 통제를 추구
→ *“이 인스턴스는 하나만 존재해야 한다.”
→ 덕분에 다른 패턴들이 만들어낸 유연성을 싱글턴이 경직시킬 수 있음..*

### 8. 더 넓은 맥락에서의 싱글턴 패턴

*“인스턴스 하나만 만드는 트릭”* → 본질을 놓치는 접근
→ *싱글턴이 진짜 다루는 것은 **“공유 자원의 관리”***

- **전통적인 싱글턴**은 ***“자원 스스로가 자신의 유일성을 관리한다”***
- **현대에서는 DI**, ***“외부 컨테이너가 자원의 수명과 유일성을 관리한다”***
- **모듈 시스템**은 ***“모듈의 캐싱 매커니즘이 자연스럽게 유일성을 보장한다”***

프론트엔드에서 싱글턴이 중요한 이유:
→ 브라우저 환경 자체가 싱글턴적 자원으로 가득하다…!

- `window`, `document`, `localStorage`, `navigator` 는 모두 유일한 전역 객체
- React의 루트 스토어, Next.js의 서버사이드 캐시, WebSocket 연결도 보통 하나만 존재..

문제는 어떤 방식으로 싱글턴을 만드냐는 것:
→ 전통적인 방식 대신, **모듈 캐싱, Context, 클로저 같은 JS 고유 매커니즘**이 그 역할을 대신..
