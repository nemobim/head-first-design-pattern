# 뽑기 기계 구조 다시 살펴보기

  - 각 상태의 행동을 별개의 클래스로 국지화했다
  - 관리하기 힘든 골칫덩어리 if 선언문들을 없앰
  - 각 상태를 변경에는 닫혀있게, GumballMachine 클래스는 새로운 상태 클래스를 추가하는 확장에는 열려 있도록 고쳤다
  - 주식회사 왕뽑기에서 제시했던 다이어그램에 훨씬 가까우면서 더 이해하기 좋은 코드베이스와 클래스 구조를 만듦

# 상태 패턴의 정의

- 상태 패턴: 객체의 내부 상태가 바뀜에 따라 객체의 행동을 바꿀 수 있다. 마치 객체의 클래스가 바뀌는 것과 같은 결과
  => 클래스가 바뀌는 것 같다? 사용하는 객체의 행동이 바뀌면 그 객체가 다른 클래스로부터 만들어진 것처럼 느껴질 것

## 상태 패턴과 전략패턴

- 상태패턴: 상태 객체에 일련의 행동이 캡슐화
- 전략 패턴: 클라이언트가 Context 객체에게 어떤 전략 객체를 사용할지 지정해줌
