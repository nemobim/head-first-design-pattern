# 고전적인 싱글턴 패턴 구현법

```java
public class Singleton {
  // 싱글톤 클래스의 하나뿐인 인스턴스를 저장하는 전역변수
  private static Singleton uniqueInstance;

  // etc instance variables

  private Singleton() {}

// getInstance 세도느는 클래스의 인스턴스를 만들어서 리턴
  public static Singleton genInstance(){
    if (uniqueInstance == null) {
      // 이 조건이면 클래스의 인스턴스가 아직 생성되지 않은 것
      uniqueInstance = new Singleton()
    }
    return uniqueInstance;
  }
  // etc methods
}
```

- 연결 풀, 스레드 풀과 같은 풀 자원 관리하는 데 사용
- 인스턴스가 하나임을 확신할 수 있는 방법?
  - public으로 지정된 생성자가 없음

# 싱글턴 패턴의 정의

- 싱글턴 패턴은 클래스 인스턴스를 하나만 만들고, 그 인스턴스로의 전역 접근을 제공
- 싱글턴 패턴을 적용할 때에는 클래스에서 하나뿐인 인스턴스를 관리하도록 만들면 됨. 그리고 다른 어떤 클래스에서도 자신의 인스턴스를 추가로 만들지 못하게 해야 함.
- 어디서든 그 인스턴스에 접근할 수 있도록 전역 접근 지점을 제공. 언제든 이 인스턴스가 필요하면 클래스에 요청할 수 있게 만들어두고 요청이 들어오면 그 하나뿐인 인스턴스를 건네주도록 설정.

singleton

# 허쉬! 초콜릿 보일러에 문제 발생

- 고정적인 싱글턴 패턴으로 코드를 고쳤음에도 ChocolateBoiler에 있는 fill() 메소드에서 아직 초콜릿이 끓고 있는데 새로운 재료를 넣고 말았음
- 멀티스레드를 사용하도록 수정하면서 발생한 이슈
- uniqueInstance 변수에 chocolateBoiler의 하나뿐인 인스턴스를 대입하면 getInstance()를 호출할 때마다 같은 인스턴스가 리턴되지 않나?

# 멀티스레딩 문제 해결하기

```java
public class Singleton{
  private static Singleton uniqueInstance;

  private Singleton(){}

  public static synchronized Singleton getInstance () {
    // synchronized를 추가해 한 스레드가 메소드 사용을 끝내기 전까지 다른 스레드는 기다려야 함. 두개의 스레드가 이 메소드를 동시에 사용하지 않도록 하기 위함
    if (uniqueInstance == null) {
      uniqueInstance = new Singleton();
    }
    return uniqueInstance;
  }
}
```

# 더 효율적으로 멀티스레딩 문제 해결하기

- 동기화 말고 다른 방법?
  방법1. getInstance의 속도가 그렇게 중요하지 않다면 그냥 둔다.
  - 다만 메소드를 동기화하면 성능이 100배 정도 저하됨.
    방법2. 인스턴스가 필요할 때 생성하지 말고 처음부터 만듦.

```java
public class Singleton {
  private static Singleton uniqueInstance = new Singleton();

  private Singleton(){}

  public static synchronized Singleton getInstance () {
    return uniqueInstance;
  }

}
```

방법3. 'DCL'을 써서 getInstance()에서 동기화 되는 부분 줄이기

- Double Checked Locking 을 사용해 인스턴스가 생성되어있는지 확인한 후 생성되어 있지 않은 경우에만 동기화
