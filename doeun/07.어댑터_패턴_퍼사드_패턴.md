# 어댑터 패턴과 퍼사드 패턴 — 인터페이스를 다루는 두 가지 방법

## 들어가며

프론트엔드 개발을 하다 보면 이런 순간들이 온다.

> "이 API 응답 왜 snake_case야... 우리는 camelCase 쓰는데."
> "회원가입 하나 하려고 API 3개를 순서대로 호출해야 한다고?"

첫 번째는 **형식이 안 맞는** 문제고, 두 번째는 **과정이 복잡한** 문제다. 비슷해 보이지만 성격이 다르다. 그리고 이 두 문제를 각각 깔끔하게 풀어주는 패턴이 있다.

- 형식이 안 맞을 때 → **어댑터 패턴**
- 과정이 복잡할 때 → **퍼사드 패턴**

이미 실무에서 무의식적으로 쓰고 있을 가능성이 높은 패턴들이다. 이번 글에서는 이 둘이 정확히 뭐가 다르고, 어디에 어떻게 쓰면 좋은지 정리해본다.

---

## 어댑터 패턴 (Adapter Pattern)

### 한 줄 개념

**맞지 않는 인터페이스를 중간에서 변환해주는 패턴.**

해외여행 갈 때 쓰는 돼지코를 떠올리면 된다. 한국 충전기와 유럽 콘센트는 모양이 다르지만, 어댑터를 끼우면 충전기를 바꾸지 않아도 그대로 쓸 수 있다. 코드에서도 마찬가지다. 외부에서 오는 데이터 형식이 내부와 다를 때, 양쪽 코드를 건드리지 않고 중간에서 변환해주는 레이어를 만드는 것이다.

### 언제 필요할까

프론트엔드에서 어댑터가 필요한 가장 흔한 상황은 **외부 API 응답을 내부 형식에 맞추는 것**이다.

백엔드가 snake_case로 응답을 주는데 프론트에서는 camelCase를 쓰고 있다면? 컴포넌트마다 `data.user_name`이라고 접근하는 건 좋은 방법이 아니다. API 응답 형식이 바뀌면 그걸 쓰는 모든 곳을 찾아서 고쳐야 하기 때문이다.

이럴 때 변환 로직을 한 곳에 모아두면, 나중에 API가 바뀌어도 어댑터 하나만 수정하면 된다.

### 코드 예시

```js
// 외부 API가 주는 형식
const externalUser = {
  user_name: "doeun",
  user_email: "doeun@example.com",
  created_at: "2024-01-15T09:00:00Z",
};

// 우리 앱에서 쓰는 형식
// { name: string, email: string, createdAt: Date }

// 어댑터 — 외부 형식을 내부 형식으로 변환
function adaptUser(externalUser) {
  return {
    name: externalUser.user_name,
    email: externalUser.user_email,
    createdAt: new Date(externalUser.created_at),
  };
}
```

"이게 패턴이라고?" 싶을 수 있다. 함수 하나잖아. 맞다. 하지만 이 변환이 컴포넌트 여기저기에 흩어져 있으면 어떨까?

```js
// ProfilePage.jsx
const name = data.user_name; // 여기서 변환하고

// SettingsPage.jsx  
const email = data.user_email; // 여기서도 변환하고

// AdminPanel.jsx
const created = new Date(data.created_at); // 또 변환하고...
```

API 응답에서 `user_name`이 `username`으로 바뀌면 세 곳을 다 찾아서 고쳐야 한다. 어댑터를 한 곳에 두면 이런 문제가 사라진다.

### 실무에서 쓰는 법

**1. API 함수에 어댑터를 붙이기**

가장 자연스러운 적용 방법이다. API를 호출하는 함수 안에서 변환까지 처리하면, 이 함수를 쓰는 쪽은 외부 형식을 전혀 신경 쓸 필요가 없다.

```js
// api/user.js
export async function fetchUser(id) {
  const { data } = await axios.get(`/api/users/${id}`);

  // 여기서 변환 완료 — 컴포넌트는 내부 형식만 알면 됨
  return {
    name: data.user_name,
    email: data.user_email,
    createdAt: new Date(data.created_at),
  };
}
```

TanStack Query와 쓴다면 `queryFn`에 이 함수를 그대로 넘기면 된다. 컴포넌트에서는 `data.name`으로 바로 접근하면 끝이다.

**2. 서드파티 서비스를 통일된 인터페이스로 묶기**

예를 들어 결제 수단이 여러 개라면, 각각의 SDK가 요구하는 형식이 다를 수 있다.

```js
const tossAdapter = {
  pay(amount, orderId) {
    return TossPayments.requestPayment({ amount, orderId, method: "카드" });
  },
};

const kakaoAdapter = {
  pay(amount, orderId) {
    return KakaoPay.ready({ total_amount: amount, partner_order_id: orderId });
  },
};

// 사용하는 쪽은 어떤 결제 수단이든 같은 방식으로 호출
function processPayment(adapter, amount, orderId) {
  return adapter.pay(amount, orderId);
}
```

토스는 `amount`를 그대로 받고, 카카오는 `total_amount`라는 키를 쓴다. 이런 차이를 어댑터가 흡수해주니까, 새 결제 수단이 추가되어도 어댑터 하나만 만들면 기존 로직은 안 건드려도 된다.

핵심은 **"변환 로직을 한 곳에 모아서, 바깥 세상의 변화가 안쪽까지 번지지 않게 막는 것"** 이다.

## 퍼사드 패턴 (Facade Pattern)

### 한 줄 개념

**복잡한 여러 단계를 하나의 깔끔한 인터페이스로 묶는 패턴.**

퍼사드(Facade)는 건물의 "정면"이라는 뜻이다. 건물 앞에서 보면 깔끔한 입구 하나지만, 그 안에는 배관, 전기, 엘리베이터 등 온갖 시스템이 돌아가고 있다. 밖에서 보는 사람은 그런 걸 알 필요가 없다. 문만 열고 들어가면 된다.

코드도 마찬가지다. 어떤 기능 하나를 실행하려고 모듈 A를 호출하고, 그 결과로 모듈 B를 호출하고, 다시 모듈 C를 호출해야 한다면 — 이 과정을 함수 하나로 묶어서 "이것만 호출하면 돼"라고 만들어두는 게 퍼사드다.

### 언제 필요할까

"하나의 기능인데 내부적으로 해야 할 일이 여러 개인 경우"가 퍼사드의 신호다.

예를 들어 회원가입은 사용자 입장에서는 버튼 한 번 누르는 거지만, 실제로는 유저 생성 → 환영 이메일 발송 → 분석 이벤트 전송 같은 여러 작업이 엮여 있다. 이걸 호출하는 쪽(컴포넌트나 핸들러)에서 일일이 관리하면 코드가 지저분해지고, 뭐 하나 빠뜨리기도 쉽다.

### 코드 예시

```js
// 각각의 서비스들
const authService = {
  createUser(data) { /* 유저 생성 API 호출 */ },
};

const emailService = {
  sendWelcome(email) { /* 환영 이메일 발송 */ },
};

const analyticsService = {
  trackSignup(userId) { /* 가입 이벤트 트래킹 */ },
};

// 퍼사드 — 여러 단계를 하나로 묶음
async function registerUser(data) {
  const user = await authService.createUser(data);
  await emailService.sendWelcome(user.email);
  analyticsService.trackSignup(user.id);
  return user;
}

// 컴포넌트에서는 이것만 호출
const newUser = await registerUser({ name: "도은", email: "doeun@example.com" });
```

`registerUser`가 퍼사드다. 이걸 호출하는 쪽은 이메일이 가는지, 트래킹이 되는지 몰라도 된다. "회원가입 해줘"라는 의도만 전달하면 나머지는 퍼사드가 알아서 처리한다.

### 실무에서 쓰는 법

**1. 앱 초기화 퍼사드**

프론트엔드 앱에 처음 진입할 때 해야 하는 일이 의외로 많다. 토큰 확인, 유저 정보 조회, 권한 로딩, 피처 플래그 가져오기 등등. 이걸 `App.jsx`에 다 풀어놓으면 코드가 길어지고, 초기화 순서가 바뀔 때마다 컴포넌트를 수정해야 한다.

```js
// initApp.js
async function initializeApp() {
  const token = getStoredToken();
  if (!token) throw new Error("인증 필요");

  const [user, permissions, features] = await Promise.all([
    fetchUserProfile(token),
    fetchPermissions(token),
    fetchFeatureFlags(),
  ]);

  return { user, permissions, features };
}

// App.jsx — 깔끔
useEffect(() => {
  initializeApp()
    .then(setAppState)
    .catch(() => navigate("/login"));
}, []);
```

나중에 초기화에 "공지사항 조회"가 추가되어도 `initializeApp` 안에서만 수정하면 된다. `App.jsx`는 건드릴 필요가 없다.

**2. 사실 커스텀 훅이 퍼사드다**

React 개발하면서 커스텀 훅을 만든 적이 있다면, 이미 퍼사드를 쓰고 있는 거다.

```js
function useAuth() {
  const { data: user } = useQuery({ queryKey: ["user"], queryFn: fetchUser });
  const loginMutation = useMutation({ mutationFn: login });
  const logoutMutation = useMutation({ mutationFn: logout });

  return {
    user,
    isLoggedIn: !!user,
    login: loginMutation.mutate,
    logout: logoutMutation.mutate,
  };
}
```

`useAuth()`를 쓰는 컴포넌트는 TanStack Query가 돌아가는지, mutation이 어떻게 설정되어 있는지 몰라도 된다. `user`, `login`, `logout`만 꺼내 쓰면 끝이다. 복잡한 구현을 간단한 인터페이스 뒤에 숨겼다는 점에서, 커스텀 훅은 React에서 가장 자연스러운 퍼사드다.

핵심은 **"사용하는 쪽이 내부 구현을 몰라도 되게 만드는 것"** 이다.

## 그래서 뭐가 다른 건데?

| | 어댑터 | 퍼사드 |
|---|---|---|
| **풀려는 문제** | 형식이 안 맞음 | 과정이 복잡함 |
| **하는 일** | A 형식을 B 형식으로 변환 | 여러 단계를 하나로 묶음 |
| **비유** | 돼지코 어댑터 | 건물 정면 입구 |
| **떠올릴 질문** | "이 데이터를 우리 형식으로 바꿔야 하나?" | "이 과정을 한 방에 처리할 수 없을까?" |

실무에서는 이 둘을 같이 쓰는 경우도 많다. 외부 API 응답을 어댑터로 변환하고, 그걸 포함한 여러 작업을 퍼사드로 묶는 식이다.

```
[외부 API] → 어댑터(형식 변환) → 퍼사드(여러 작업을 하나로) → 컴포넌트
```

---

## 마무리

어댑터 패턴과 퍼사드 패턴은 둘 다 "인터페이스를 정리하는" 패턴이지만, 바라보는 문제가 다르다.

- **어댑터**는 "형식이 안 맞는 것"을 해결한다. 외부와 내부 사이에 변환 레이어를 두어서, 한쪽이 바뀌어도 다른 쪽에 영향이 안 가게 막는다.
- **퍼사드**는 "과정이 복잡한 것"을 해결한다. 여러 단계를 하나의 함수(또는 훅)로 묶어서, 사용하는 쪽의 부담을 줄인다.

API 응답 변환 함수를 만들어본 적 있다면 어댑터를 쓴 거고, 여러 작업을 하나의 함수나 훅으로 묶어본 적 있다면 퍼사드를 쓴 거다. 이름을 붙이고 의식적으로 분리하기 시작하면, "여기는 어댑터니까 변환만 하자", "여기는 퍼사드니까 흐름만 관리하자"처럼 역할이 명확해진다. 코드를 읽는 사람도 의도를 빠르게 파악할 수 있고, 수정할 때도 어디를 건드려야 하는지 바로 보인다.
